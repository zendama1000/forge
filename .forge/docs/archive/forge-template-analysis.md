# Forge テンプレート横断分析：品質を落としている共通パターン

## 分析の前提

- 対象：`.forge/templates/` 全8ファイル + `.claude/agents/` 全7ファイル + オーケストレーション（research-loop.sh, common.sh）
- ユーザー報告の症状：(1) Researcherの情報収集が浅い・的外れ (2) 最終レポートが「で、何すればいいの？」感
- 外部知見：Anthropic公式プロンプトエンジニアリングガイド、GPT Researcher/Deep Research Agent設計、MASS論文(2025)、Anthropic Context Engineering記事

---

## 発見した共通パターン（7件）

### P1: 「形」の制約にトークンを使いすぎ、「思考の質」への投資が足りない

**該当テンプレート：全8ファイル**

全テンプレートが末尾に「出力形式（厳守）」セクションを持ち、JSONスキーマを詳細に定義している。これ自体は正しい設計だが、問題はバランス。

**定量的な観察：**

| テンプレート | 全体行数 | 出力形式定義の行数 | 思考ガイダンスの行数 | 形式:思考の比率 |
|---|---|---|---|---|
| researcher-prompt.md | 40 | 22 (55%) | 3 (7%) | **7.3:1** |
| scope-challenger-prompt.md | 55 | 20 (36%) | 5 (9%) | **4.0:1** |
| synthesizer-prompt.md | 45 | 20 (44%) | 4 (9%) | **5.0:1** |
| devils-advocate-prompt.md | 70 | 28 (40%) | 5 (7%) | **5.6:1** |
| criteria-generation.md | 60 | 30 (50%) | 6 (10%) | **5.0:1** |
| task-planning-prompt.md | 90 | 45 (50%) | 8 (9%) | **5.6:1** |

**なぜ問題か：**
- Anthropic公式ガイドによれば、CoT（Chain of Thought）はClaudeの分析・推論タスクの品質を「劇的に」改善する
- 現テンプレートはCoTを一切使っていない。「〜してください」という指示だけで、「どう考えるか」の道筋を示していない
- JSON直接出力を強制しているため、Claude自身の推論プロセスが発生しない（出力がそのまま最終形式 = 思考なしで答えを出す構造）

**具体例（researcher-prompt.md）：**
```
現状のタスク指示:
1. 上記の問いに回答する情報をWeb検索で収集する    ← 「どう検索するか」が無い
2. 発見を構造化して報告する                       ← 「どう構造化するか」が無い
3. 他の視点のことは考慮しない                     ← 制約のみ
```

Researcherが浅い情報しか集められない直接的な原因がここにある。検索戦略・深掘り判断・情報品質評価のガイダンスがゼロ。

---

### P2: few-shot example（良い出力例）が1つもない

**該当テンプレート：全8ファイル**

Anthropicの公式ベストプラクティスで最も強調されているテクニックの1つが「multishot prompting」— 良い出力の具体例を示すこと。現テンプレートはJSONスキーマ（構造の定義）はあるが、「良い出力がどういう内容か」の例が一切ない。

**スキーマ vs 例の違い：**
```
スキーマ（現状やっていること）:
  "answer": "発見の要約"      ← 「要約」のレベル感が不明

例（やっていないこと）:
  "answer": "React Server Componentsは2024年時点でNext.js 14+のApp Routerでのみ
   プロダクション対応。Remixは2025Q1にRSC対応予定（公式ブログ2024-11）だが
   リリース遅延の前例あり。Astroは独自のIslandsアーキテクチャで代替アプローチ。"
```

前者ではLLMは「要約」の粒度を自分で判断する（→浅くなる）。後者では期待される具体性・情報量が伝わる。

**影響の大きさ：**
- Researcher：何をもって「十分な調査」かわからない → 浅い検索で満足
- Synthesizer：何をもって「統合」かわからない → 寄せ集めになる
- SC：何をもって「良い問い」かわからない → 的外れな分解になりうる

### P3: JSON直接出力の強制がCoT（推論過程）を殺している

**該当テンプレート：全8ファイル**

```
現在の出力指示:
「レスポンスの最初の文字は `{`、最後の文字は `}` であること」
```

この制約は validate_json() の3層リカバリ（common.sh）で必要とされているが、代償が大きい。

**メカニズム：**
1. 最初の文字が `{` → Claudeは思考プロセスなしで直接JSON生成を開始
2. JSON構造に意識が向く → 内容の質より形式の遵守が優先される
3. 自己修正の機会がない（「待てよ、この推奨は矛盾している」と気づいても、JSONの途中では修正できない）

**外部知見との比較：**
- GPT Researcher: planner → execution → publisher の3段階で、中間段階は自然言語
- Deep Research Agent (Egnyte): 各サブエージェントがまず自然言語で推論し、最終段階でのみ構造化
- Anthropic公式: 「CoTなしで出力すると、思考プロセスが発生しない」と明記

**特にダメージが大きいテンプレート：**
- `devils-advocate-prompt.md`：批判的思考こそCoTが最も有効なのに、いきなりJSON
- `synthesizer-prompt.md`：矛盾発見・統合判断にはまず「考える」フェーズが必要
- `scope-challenger-prompt.md`：問いの設計には探索的思考が必要

### P4: タスクの詰め込みすぎ（特にDA、Synthesizer）

**該当テンプレート：devils-advocate-prompt.md, synthesizer-prompt.md, scope-challenger-prompt.md**

Anthropic公式ガイドの推奨：「1つのプロンプトに1つの焦点タスク。複数の変換やステップがある場合、チェイニングで分割すべき」

**devils-advocate-prompt.mdのタスク数：**
```
0. 前回must_fixの修正状況を検証する
1. 推奨の前提を攻撃する
2. 確証バイアス、生存者バイアス等を検出する
3. 最悪シナリオを描く
4. 機会費用を算出する
5. 調査スコープが適切か検証する
+ verdict判定
+ 前回フィードバック検証（previous_feedback_review配列）
+ 構造化must_fix出力（id, category, description, resolution_criteria, related_perspectives, carry_count）
```

これは1回のLLM呼び出しに対して9つ以上の独立した分析タスクを要求している。

**比較：**
- 元のRalphアプローチ：1ループ = 1タスク = 1フォーカス
- GPT Researcher: 各エージェントは1つの質問に対して1つの回答
- MASS論文の知見：ブロックレベルの最適化がまず重要（各エージェントのプロンプトを個別に最適化してからトポロジーを考える）

**結果：**
- DAが毎回違うことを言う（re-execution lottery）のは、9タスク中どれにリソースを割くかがランダムになるため
- すべて中途半端に触れて、どれも深くならない

### P5: Researcherの検索戦略がLLM任せ

**該当テンプレート：researcher-prompt.md, researcher.md（agent定義）**

現在のResearcherへの指示全体：
```
agent定義: 「Web検索を積極的に使用する（情報収集が本務）」
テンプレート: 「上記の問いに回答する情報をWeb検索で収集する」
```

これだけ。以下が完全に欠落している：

**欠落している検索ガイダンス：**
1. **検索戦略**: 広い検索→絞り込み検索のパターン、複数キーワードの試行
2. **ソース品質評価**: 公式ドキュメント > ブログ > フォーラム等の優先順位
3. **深掘りの判断基準**: 「1回の検索で満足せず、矛盾する情報や反例も探す」
4. **情報の鮮度基準**: 「2年以上前の情報には注意フラグを付ける」
5. **検索回数の目安**: 「1つの問いにつき最低3-5回の検索を行う」
6. **検索失敗時の対応**: 「見つからない場合のフォールバック（関連キーワード変更、英語での再検索等）」

**外部比較：**
- GPT Researcher: 各質問に対して専用のcrawlerエージェントが複数ソースを収集し、要約とソース追跡を行う
- Deep Research Agent: 検索ワーカーは「検索の実行に特化」し、親エージェントが検索戦略を計画する分離設計
- Anthropic Context Engineering記事: 「各エージェントに明確で集中した責務を与え、曖昧さを減らす」

**Sonnetモデルへの影響：**
research.jsonのデフォルト設定でResearcherはsonnetモデルを使用。Sonnetは指示が具体的なほど品質が上がるモデルであり、曖昧な指示への耐性がOpusより低い。検索戦略の欠落はSonnetに特にダメージが大きい。

### P6: Synthesizerの入力が肥大化しすぎ

**該当テンプレート：synthesizer-prompt.md**

`{{ALL_REPORTS}}` に全Researcherレポート（4-6視点分）がそのまま注入される。

**コンテキスト消費の試算：**
```
1視点のResearcherレポート: 約2,000-4,000トークン
× 6視点 = 12,000-24,000トークン（ALL_REPORTSだけで）
+ investigation-plan.json: 約1,000トークン
+ decisions: 約500-2,000トークン
+ DA_FEEDBACK（2周目以降）: 約1,000-2,000トークン
+ テンプレート本体: 約800トークン
─────────────────────
合計: 15,000-30,000トークン（入力だけで）
```

**なぜ問題か：**
- Anthropic Context Engineering記事: 「コンテキストウィンドウ内の情報の関連性の問題は、ウィンドウサイズが大きくなっても残る」
- 情報量が多いほど、モデルの注意が分散し「寄せ集め」統合になりやすい
- Anthropic Long Context Tips: 「最も重要な情報を最初か最後に配置する」— 現在は視点の出現順（alphabet順 or 実行順）

**GPT Researcherとの比較：**
- GPT Researcher: 各ソースを個別に要約してからpublisherに渡す（中間要約層あり）
- 現Forge: 生データを丸ごとSynthesizerに投げる（要約なし）

### P7: 「で、何すればいいの？」問題 — アクショナブルな出力への導線が弱い

**該当テンプレート：synthesizer-prompt.md, research-loop.sh (generate_final_report)**

最終レポートの「で、何すればいいの？」感の原因は複合的：

**Synthesizerの推奨フォーマット：**
```json
"recommendations": {
  "primary":  {"action": "...", "rationale": "...", "risks": ["..."]},
  "fallback": {"action": "...", "rationale": "...", "trigger": "..."},
  "abort":    {"rationale": "...", "opportunity_cost": "..."}
}
```

問題点：
1. `action` フィールドに具体性の要求がない（「Reactを採用する」レベルで終わりがち）
2. 「次のステップ」「タイムライン」「必要なリソース」「判断に必要な追加情報」がスキーマに存在しない
3. 3段階推奨（primary/fallback/abort）は構造として良いが、**判断基準**が抽象的になりやすい

**最終レポート生成の問題：**
```bash
# research-loop.sh generate_final_report()
local prompt="以下のリサーチ結果から、人間が読みやすい日本語の最終レポートを生成してください。
Markdown形式で、見出し・表・箇条書きを適切に使ってください。"
```

このプロンプトには：
- レポートの構造指定がない
- 「意思決定者が判断できる」品質基準がない
- 読者の前提知識レベルの想定がない
- 「結論ファースト」等の構成指示がない

つまり、リサーチの集大成であるレポート生成が、全テンプレート中で最も雑なプロンプトで実行されている。

## パターン間の因果関係

```
P5 (検索戦略なし)
  → Researcherが浅い情報しか集められない
    → P6 (入力肥大) + P2 (例なし)
      → Synthesizerが「寄せ集め」統合をする
        → P3 (CoTなし) + P4 (タスク詰め込み)
          → DAが表面的な指摘しかできない / 毎回違うことを言う
            → P7 (アクションなし)
              → 最終レポートが「で、何すればいいの？」

P1 (形式偏重) は全段階に横断的に影響
```

**根本は上流（P5→P2→P3）に集中している。** DAやレポートの問題は、上流の品質が低いことの反映でもある。

---

## 改善の方向性（選択肢として）

改善策は大きく3つのアプローチに分かれる。排他的ではない。

### A. 最小変更アプローチ（テンプレート修正のみ）
- 各テンプレートにCoTセクション追加（「まず〈thinking〉タグ内で分析し、その後JSON出力」）
- researcher-promptに検索戦略ガイダンス追加
- 1-2個のfew-shot例を追加
- validate_json()のリカバリを拡張してthinkingタグ付き出力に対応

**メリット:** オーケストレーション変更なし。即座に試せる
**デメリット:** P4(タスク詰め込み)とP6(入力肥大化)は未解決

### B. 中程度の変更（テンプレート + 一部オーケストレーション）
- Aに加えて、Synthesizer前に中間要約ステップを挿入
- DAのタスクを分割（分析フェーズ → 判定フェーズの2段階）
- 最終レポート生成テンプレートを専用テンプレート化

**メリット:** P4/P6も改善。品質への影響が大きい
**デメリット:** research-loop.shへの改修が必要。コスト微増（ステップ追加分）

### C. アーキテクチャ寄りの変更
- Researcher内部でiterativeな検索ループ（GPT Researcher方式）
- 各ステージの出力に自己評価メカニズム追加
- DAを独立した2-passレビュー（broad scan → deep dive）に

**メリット:** 根本的な品質改善
**デメリット:** 大幅な改修。テスト工数大。コスト増

## 補足：コストの観点

「コストがかかる割に品質が悪い」の「コスト」も分解しておく：

| コスト要因 | 現状の構造的問題 |
|---|---|
| Researcher × 6視点（毎回） | 固定4視点が全テーマに強制適用。テーマによっては2-3視点で十分 |
| CONDITIONAL-GOループ（最大3周） | DAのre-execution lotteryで無駄な周回が発生 |
| 全レポート丸投げ（Synthesizer） | 要約なしの大量入力 → 高トークン消費 |
| 最終レポート生成 | テンプレートなしの曖昧プロンプト → 品質低い割にトークン消費 |

品質改善とコスト削減は一部両立可能（P5改善でResearcherの空振り減少、P4改善でDAループ削減）。
