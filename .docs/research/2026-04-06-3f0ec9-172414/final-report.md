## 3. 視点別の主要発見

### 3.1 技術的実現性（technical）

#### サブエージェント2層構成: 成立する

- SKILL.md → phase-agent（Agent tool）→ Read toolでJSONプロファイル読込の構成は機能する
- **非再帰制約**（サブエージェントがさらにサブエージェントを生成できない）はJSONプロファイル設計で正しく回避済み
- 各サブエージェントは独立した200K（有料Opus/Sonnetは1M）トークンウィンドウを持つ
- プロファイルが巨大な場合（100KB超）はウィンドウ圧迫リスクあり

#### phase-state.json排他書込: 高リスク

- 並列Agent tool実行時の破損リスクは **実証済みの既知問題**（Claude Code自身の.claude.json破損issue多数）
- SKILL.mdのシリアライズ指示の遵守率は約70%で、100%の保護にならない
- **必要な対策**: PostToolUseフックでのアトミック置換（temp file → rename）強制

#### SKILL.md指示 vs Hooks の信頼性差

| 制御方式 | 遵守率 | 適用場面 |
|----------|--------|----------|
| SKILL.md指示（プロンプト） | 70〜90% | ガイダンス・推奨事項 |
| Claude Code Hooks | **100%** | 必ず実行すべき処理 |

> **結論**: ファイル書込の排他制御・ハッシュ計算・バリデーション再試行など「必ず実行しなければならない」処理はHooksに移行すべき

### 3.3 リスク・失敗モード（risk）

#### 4つのリスクは連鎖的に作用する

```
(1) 曖昧ヒアリング → desire-architecture誤生成
      ↓
(2) 18フィールドJSONの意味的品質劣化で固定化
      ↓
(3) 倫理ゲートは変形表現に構造的脆弱性（カバレッジ85〜93%）
      ↓
(4) Phase遷移品質ゲート不在 → 劣化が下流5ドキュメントに無検証で伝播
      ＝ 最大のシステムリスク（エラー増幅率 最大17.2倍）
```

#### Phase 1 生成品質リスク

- LLMは曖昧・矛盾入力に対し「もっともらしい混合出力」を生成する傾向（PRIN現象）
- 3段バイアス補正はRBD型パイプラインで精度+18.5%改善の報告があるが、補正自体がバイアスを導入するリスクも
- **confidence: medium**（desire-architecture特化の誤り率データなし）

#### 倫理ゲート漏れリスク

| 層 | 方式 | カバレッジ | 脆弱性 |
|----|------|-----------|--------|
| 第1層 | 静的パターンマッチ（20件） | 20〜40%（変形表現に対して） | 言い換え・婉曲表現 |
| 第2層 | LLMジャッジ | F1=0.805〜0.939 | 日本語多言語ギャップ |
| **2層合計** | — | **85〜93%**（意図的迂回なし） | 占い文脈特有の断定スペクトル |

#### Phase遷移品質ゲート不在

- 全多エージェント失敗の21.3%がTask Verification問題に起因（うち不完全検証8.2%）
- 対策として3層検証（スキーマ検証+コンテンツバリデーション+独立ジャッジ）が推奨
- PwC事例: 多層バリデーション導入で精度10%→70%に7倍改善

### 3.4 実装パターン比較（alternatives）

#### 各問いの最適選択肢

| 問い | 最適選択肢 | 理由 |
|------|-----------|------|
| JSON要約抽出方式 | (B) テンプレート埋込み + フック検証 | CLI -pモードではtool_use不可。フック追加で(C)に匹敵 |
| Phase遷移品質ゲート | (B) JSONスキーマバリデーション追加 | 決定論的・ミリ秒・追加LLMコストゼロ。「機械的判定」ロック内最適 |
| カスケード検知 | (B) PostToolUseフック | 信頼性100%・設定のみ・追加プロセス不要 |
| 倫理ゲート第1層 | ハイブリッド: (B)grep/jq + (A)LLMフォールバック | 機械的高速除外 + 意味的ジャッジの組合せ |
| ジャンルプロファイル注入 | (C) 必要セクションのみ抽出 | トークン50%以上削減。小規模なら(A)全文でもOK |

> **共通パターン**: 「機械的・決定論的な実装」がロック決定「機械的判定」の精神に合致。LLM依存の柔軟実装はいずれも信頼性面でリスクが高い。

### 3.5 理論忠実性（domain_fidelity）

#### Step 1→4 線形構造: 理論に準拠（問題なし）

- 理論第3章は「壁は上から順に潰す」を**鉄則**と明示
- 「壁2と壁3が同時に作用する」非線形パターンは理論が否定するもの
- むしろ**現テンプレートの並列表示が理論違反**であり、Step 1→4改修は忠実度を向上させる

#### 活性化強度スコアリング: 理論根拠に問題あり

| 項目 | 現設計 | 理論の実態 |
|------|--------|-----------|
| 基準 | 「クロスファンクション表に登場 かつ 3フェーズ以上カバー」 | 「3フェーズ以上」は理論第5章に記述なし |
| 優先度 | 高/中/低の3値を独自設計 | 理論に既存の「最優先/高/中」が定義済み |
| 導出 | Chapter 2→5の跨ぎ推論 | Chapter 5はフェーズを参照していない |

> **推奨**: 理論第5章のクロスファンクション表の既存優先度を一次基準として採用

#### 欲求類型の術語ドリフト: 進行中

| 理論定義（正） | ドリフト例（誤） |
|---------------|----------------|
| ①不確実性の解消 | 所有欲求 |
| ②自己理解 | 体験欲求 |
| ③承認・肯定 | 承認欲求 |
| ④変容 | 安心欲求 |

- JSON要約スキーマに**enum制約**（`uncertainty_resolution` | `self_understanding` | `validation_affirmation` | `transformation`）を定義し機械的に防止すべき

#### 情報損失リスク5件

1. 壁①の3層構造（表層/中層/深層）がJSONフィールド1値に圧縮
2. 設計警告テキスト（負の副作用記述）がJSON化で消失
3. クロスファンクション手法の「負の副作用」が`bool`フラグだけでは表現不能
4. ジャンル固有の「欲求の現れ方」テキストがactivation_types配列に収容不可
5. 「表層入口→深層充足」の動的プロセス記述が静的スナップショットに

### 3.6 UX一貫性（ux_coherence）

#### Phase 1 ヒアリング: Q3が最大のウィークポイント

| 質問 | 種別 | リスク |
|------|------|--------|
| Q1 ジャンル | 自己参照型 | 低 |
| Q2 ターゲット | 自己参照型 | 低 |
| **Q3 迷う理由** | **他者視点推測型** | **高** — 占い師はマーケ心理的障壁の言語化が困難 |
| Q4 強み | 自己参照型 | 低 |

- Blind Spot Bias: サービス提供者は顧客の認知バイアスを過小評価しやすい
- **改修案**: 「お客様から最も多く聞く不安や心配は何ですか？」（経験ベースの事実報告に変換）

#### Progressive Disclosure: 質問数上限が必要

- UX業界ガイドライン: 追加質問は**最大2〜3問**が上限（3層超は逆効果）
- 3段バイアス補正が全発動した場合: **最悪6〜9問追加** → ガイドライン大幅超過
- チャットボット離脱率: 予期しない体験1回目で14.9%、2回連続で25.9%
- **必須**: 各補正質問に「なぜ聞くのか」の1行説明

#### 専門概念の平易化

- 「desire-architecture」「壁の解体戦略」「欲求活性化マッピング」は非技術者には理解困難
- 平易な翻訳責務は**phase2-agent（LLM）**が担うべき（静的ドキュメントでは不十分）
- 5ドキュメント生成中はチェックリスト型プログレスインジケーターで進捗を可視化

## 5. 視点間の矛盾と解決

### 5.1 JSON要約抽出方式（alternatives vs technical）

- **矛盾**: alternativesはtool_use構造化出力(C)を最優位と評価するが、technicalはCLI -pモードでは使用不可と指摘
- **解決**: CLI前提では**(B)テンプレート埋込み + PostToolUseフック検証**が実質最適解。API経由の将来拡張時に(C)に移行する余地を残す

### 5.2 phase-state.json破損（technical × risk）

- **複合リスク**: 排他書込破損 + Phase遷移品質ゲート不在 → 破損→誤判定→連鎖劣化の最悪シナリオ
- **解決**: 2段防御 — (1)フックでアトミック置換強制 (2)遷移判定時にスキーマバリデーション + バイト数チェック

### 5.3 バイアス補正質問数（ux_coherence vs d-115359§3）

- **矛盾**: 設計は3段補正だが、UX上限は追加2〜3問
- **解決**: 発動上限を2段に制限。第3段は不確実性スコアが閾値超過時のみ
