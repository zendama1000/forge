## 3. 視点別調査結果

### 3.1 技術的実現性（Technical）

#### 最重要発見: 3段委譲は技術的に不可能

Claude Code はサブエージェントがさらにサブエージェントを生成することを**公式に禁止**している。

- サブエージェントは Agent/Task ツールを持たない（GitHub issue #19077, #4182, #32731 で確認）
- SKILL.md → phase-agent の1段委譲は `context: fork` で可能
- phase-agent → genre-agent の2段目は**ネイティブに不可能**
- `claude -p` による迂回は OOM クラッシュ・可視性ゼロのリスクあり

#### その他の技術的知見

| 項目 | 結論 | 信頼度 |
|------|------|--------|
| 状態ファイル競合 | アトミック書き込みまたはシリアライズパターンが必須。Claude 自身の `claude.json` 破損バグ（#29051）で実証済み | 高 |
| Markdown→JSON 抽出精度 | JSON Schema + 低温度(0.0-0.1) + 自己検証指示で 95-99% 達成可能 | 高 |
| コンテキスト制約 | 1M トークン対応（GA済み）で実質問題なし。6テンプレート+理論+倫理+状態は数万トークン程度 | 高 |

#### 技術的ギャップ

- `context: fork` が Skill ツール経由で安定動作するか未確認（#17283 未解決）
- Agent Teams 機能の代替可能性が未検証

### 3.3 リスク・失敗モード（Risk）

#### 特定された5つの失敗モード

| # | 失敗モード | 深刻度 | 対策 |
|---|-----------|--------|------|
| 1 | Phase 1 品質低下 → Phase 2 修正負荷の指数増大 → ユーザー離脱 | 高 | Phase 境界に自動品質ゲート + Self-Refine パターン |
| 2 | 静的7パターンの偽陰性（暗示的・文脈依存表現を見逃す） | 高 | 多層防御: 正規表現 + LLM 文脈理解 + 人間レビュー |
| 3 | JSON 要約抽出失敗（パース不能・フィールド欠落） | 中 | 多層リトライ → LLM 自動修復 → デフォルト値 |
| 4 | カスケード検知漏れ → ドキュメント間不整合 | 高 | OWASP ASI08 2026 で正式脅威分類。トレース + バージョン管理 |
| 5 | LLM-as-judge 判定揺れ（同一コンテンツで PASS/FAIL 変動） | 中 | 複数回評価 + バイナリ設計 + 明確なルーブリック（目標: Cohen's κ > 0.8） |

#### 判定揺れ対策の詳細

- 温度=0 でも確率的揺れは消えない
- 異なる LLM ジャッジ間の合意率は約 80%（20%不一致）
- スワップ拡張・複数回評価・バイナリ PASS/FAIL 設計が推奨
- 倫理判定は独立ゲートとして扱い、他スコアとの平均化は禁止

### 3.4 代替案・競合（Alternatives）

#### 5つの比較分析の結論

**1. ジャンル別ロジック: サブエージェント vs JSON プロファイル**

| 基準 | サブエージェント | JSON プロファイル |
|------|----------------|-----------------|
| 適合条件 | ツール・権限がジャンル間で本質的に異なる | プロンプトのトーン・スタイルレベルの差異 |
| 品質 | ツール5本専用で 23〜31% 品質向上 | ツール数10〜15超で精度低下 |
| コスト | トークン 4〜220倍増 | 最小限 |
| **結論** | 占い系は主にプロンプト差異 → **JSON プロファイルが効率的** |

> Princeton 研究: ツール・文脈同一条件下で単一エージェントがマルチエージェントを 64% のタスクで上回る

**2. テンプレート間パイプライン: JSON vs Markdown vs YAML/TOML**

| 基準 | JSON | Markdown | YAML | TOML |
|------|------|----------|------|------|
| 機械パース信頼性 | **最優位** | 低 | 中 | データ不足 |
| LLM 入力精度 | 中 | 中 | **最高** | データ不足 |
| トークン効率 | 基準 | +16% 節約 | 同等 | データ不足 |
| **推奨** | エージェント間データ受け渡しには **JSON** |

**3. カスケード検知: 3案比較**

| 方式 | 信頼性 | 実装コスト | 推奨 |
|------|--------|-----------|------|
| SKILL.md 指示のみ | 最低（17.2倍エラー増幅） | 最低 | 単独使用不可 |
| ファイルハッシュ検知 | 高（決定論的） | 中 | ハイブリッドの一部として |
| 明示的依存グラフ | 最高（正確な伝播範囲） | 高 | ハイブリッドの一部として |
| **推奨** | **ハッシュ + 依存グラフのハイブリッド** |

**4. Phase 1 内部マッピング: ルールベース vs LLM vs ハイブリッド**

| 方式 | 長所 | 短所 |
|------|------|------|
| 完全ルールベース | 高速・低コスト・決定論的 | 新規パターンに脆弱 |
| 完全 LLM ベース | 文脈理解・曖昧性処理 | 高コスト・低速 |
| **ハイブリッド** | **精度とコストを両立** | ルーター閾値設計が必要 |

**5. 倫理ゲート第1層: 3方式比較**

| 方式 | 速度 | 精度 | 推奨 |
|------|------|------|------|
| 正規表現マッチ | 最速 | 既知パターンのみ | ファストパスとして |
| 禁止語辞書+文脈チェック | 中 | やや改善 | 中間層として |
| 軽量分類モデル（Haiku） | 低速 | 高精度 | 精度パスとして |
| **推奨** | **正規表現（ファストパス） + Haiku（精度パス）の2段構成** |

> Anthropic 自身が Haiku を安全スクリーニング第1層として使用している実績あり

### 3.5 購買構造理論の実装忠実性（Domain Fidelity）

#### 4層の忠実性評価

| 層 | 忠実性 | 状態 |
|----|--------|------|
| テンプレート（desire-architecture-template.md） | **高** | 欲求2層構造・壁の順序・壁①の3層構造を正確に反映。ただし壁の依存関係が並列セクション表示（非可換性の欠如） |
| エージェント（旧3体） | **ゼロ** | concept-hearer/worldview-generator/profile-generator は購買構造理論を一切参照しない。全面作り直し対象 |
| 内部マッピング（Phase 1） | **未設計** | ユーザー入力 → 欲求4類型/壁4タイプへの変換ロジックが未定 |
| JSON パイプライン | **未設計** | スキーマ未定義。圧縮時の情報損失リスクが未対処 |

#### JSON 要約圧縮時に失われるリスクのある理論情報（5種）

1. **壁の依存順序**（①→②→③→④の非可換制約）— 配列の順序制約がないと消失
2. **壁①の3層構造**（表層/中層/深層）— 単一スコアに圧縮されると対処手法の区別が消える
3. **クロスファンクション手法**（両軸に効く手法の特定）— 汎用施策リストに平坦化されやすい
4. **設計注意点テキスト**（「覚悟の教育は壁④を高くする」等）— JSON フィールドでなければ消失
5. **フェーズ別欲求活用テーブル**（集客/体験/リピート/導線）— 「主欲求」1フィールドへの圧縮では不十分

#### ジャンル別壁プロファイルの根拠

- 理論文書は **Version 1.0 (Draft)** であり、ジャンル別の壁の厚さは**実証データなしの仮説ベース**
- ジャンル間の障壁差を定量的に比較した学術研究は**存在しない**
- テンプレートは `{高/中/低}` プレースホルダーを採用しており、LLM 推論への委譲が設計意図

#### 活性化強度の評価基準

- 「活性化強度（高/中/低）」の評価基準は理論文書に**存在しない**
- 理論第5章のクロスファンクション表から導出可能だが、理論著者の意図との一致は未確認
- **明示的なスコアリングルールの定義が必要**

### 3.6 対象ユーザー体験の整合性（UX Coherence）

#### 構造的 UX リスク（4つ）

**1. CLI 媒体の採用障壁**

- CLI は非技術者にとって根本的な障壁（コマンド記憶・構文理解・視覚フィードバック欠如）
- Progressive Disclosure の研究はほぼ全て**視覚的 UI を前提**
- 占い・スピ系事業者の 78% がリモートサービス提供（デジタル化は進行中）だが、SaaS の GUI 利用であって CLI 習熟ではない

> **緩和策**: Claude の対話型 CLI はコマンド入力型とは異なる。日本語での丁寧なフェーズ説明・テキストアンカー（`=== Phase 1 / 3 ===`）で GUI 的体験に近づけられる余地あり

**2. Phase 1 の自己認識バイアス**

- Q3「お客様が申し込みを迷う最大の理由」は**自己奉仕バイアス**が高確率で作用
- 事業主は成功要因を自己帰属し失敗は外部帰属する傾向
- 顧客自身も不満の根本原因を正確に言語化できないケースが多く、事業主による二次推定はさらに精度低下

> **緩和策**: 明確化質問の追加・反証例の提示・具体的顧客エピソードへの深掘り

**3. 6ドキュメント修正判断の認知負荷**

- 「ドキュメント A を変えると下流 B にも影響する」という依存関係の理解は非技術者には本質的に難しい
- 各ドキュメントの役割説明 + 修正内容からの逆引きマッピング + 修正すべきでない箇所の明示が必要

**4. カスケード通知のユーザー混乱**

- 開発者コミュニティでもカスケードの意味論的混乱が確認されている
- 非技術者向けには自動適用でなく**個別確認ステップ**が必須
- 「上流」「下流」等の専門用語を使わない平易な言い換えが有効

## 4. 視点間の矛盾と解決策

### 矛盾1: ジャンル別サブエージェント委譲の技術的不可能性

| 対立 | Technical vs ロック決定 (d-20260406-081021) |
|------|---------------------------------------------|
| 内容 | ロック決定は「Phase別3体 + ジャンル別サブエージェント委譲のハイブリッド構成」を規定。Technical はサブサブエージェント生成の公式禁止を確認 |
| **解決** | **ジャンル固有ロジックを JSON プロファイルとして `.claude/agents/genres/` に配置**。phase-agent が SKILL.md の prompt パラメータでプロファイルパスを受け取り Read ツールで読み込み。ロック決定の意図（ジャンルロジック分離）と「同時起動は常に3体以下」制約を維持 |

### 矛盾2: JSON 要約のパース信頼性 vs 理論情報の完全性

| 対立 | Alternatives vs Domain Fidelity |
|------|--------------------------------|
| 内容 | Alternatives は JSON が機械パースで最優位と結論。Domain Fidelity は 5種の理論的重要情報の圧縮損失リスクを指摘 |
| **解決** | **「リッチ JSON 要約」方式を採用**。壁の依存順序を配列順序制約として、クロスファンクション手法を関係性フィールドとして、フェーズ別欲求活用をネスト構造として明示的にスキーマに含める |

### 矛盾3: Phase 1 入力品質のバイアス × 修正負荷の増大

| 対立 | UX Coherence × Risk（複合的失敗モード） |
|------|---------------------------------------|
| 内容 | バイアス含みの入力 → 歪んだ desire-architecture → Phase 2 大量修正 → ユーザー離脱 |
| **解決** | **Phase 1 に3段バイアス補正機構を組込**。(1) ジャンル別典型壁パターン照合、(2) 具体的顧客エピソード深掘り、(3) マッピング結果の平易な確認提示。Progressive Disclosure 設計で不確実性が高い場合のみ発動 |

### 矛盾4: コスト最適化 vs 信頼性最適化

| 対立 | Cost vs Alternatives |
|------|---------------------|
| 内容 | Cost はプロンプトキャッシュで直接参照のコスト増 90% 抑制可能と指摘。Alternatives は JSON 要約のパース信頼性を推奨 |
| **解決** | **JSON 要約パイプライン採用** + 共通参照部分（purchase-structure-theory.md の定数的セクション）にプロンプトキャッシュを活用するハイブリッドコスト戦略 |

### 矛盾5: Phase 遷移の機械的一貫性 vs ユーザー理解

| 対立 | Technical vs UX Coherence |
|------|--------------------------|
| 内容 | Technical はアトミック書き込み/シリアライズが必須。UX Coherence は平易なフェーズ遷移通知が必須 |
| **解決** | **Phase 遷移制御を2層に分離**。内部層: SKILL.md のみが phase-state.json に書き込み（シリアライズパターン）。表示層: 読み取り後にユーザー向け進捗メッセージを生成 |

## 5. ロック済み決定事項との整合性

### 整合（8項目中6項目）

| ロック決定 | 評価 |
|-----------|------|
| SKILL.md ベース実装形態 | 完全整合。全6視点が前提として分析 |
| 5ドキュメント構成と依存順序 | 完全整合。理論忠実性・カスケード検知の両面で検証 |
| purchase-structure-theory.md の理論的基盤 | 完全整合。テンプレート層の忠実性を確認 |
| 3フェーズフロー（Phase 1→2→3） | 完全整合。Phase 境界品質ゲート・遷移案内を設計 |
| 倫理ゲート二段構成 | 完全整合。Anthropic 実装例で支持。Haiku でコスト 0.4〜2% |
| 旧コード削除タイミング・素材再利用 | 完全整合。先行削除推奨ゼロ |

### 要修正（2項目）

| ロック決定 | 問題 | 対応 |
|-----------|------|------|
| ジャンル別サブエージェント委譲 | 3段委譲が技術的に不可能 | JSON プロファイル参照方式へ変更（意図は維持） |
| Phase 1 ヒアリング質問設計 | Q3 に自己奉仕バイアスの歪みリスク | バイアス補正の後続プロンプト追加（質問自体は変更なし） |

## 6. 統合所見と推奨アクション

### 総合判断

ロック済み設計の大枠は妥当。6視点の横断分析で浮上した **3つの構造的課題**（3段委譲不可能・JSON 圧縮損失・Phase 1 バイアス）と **2つの設計空白**（活性化強度基準・カスケード検知メカニズム）は、いずれも設計レベルで解消可能。

### 推奨: 5つの実装仕様決定を行い開発フェーズへ進む

#### (1) ジャンル別ロジックの JSON プロファイル化

`.claude/agents/genres/` にジャンル別 `.json` プロファイルを配置（サブエージェント `.md` ではなく）。

```json
{
  "genre_id": "tarot",
  "wall_thickness_guide": {
    "wall1": { "tendency": "高", "layers_detail": "..." },
    "wall2": { "tendency": "中", "..." : "..." }
  },
  "desire_activation_patterns": [
    {
      "technique": "...",
      "primary_desire": "self_understanding",
      "secondary_desires": ["approval"],
      "cross_function": true
    }
  ],
  "prompt_fragments": {
    "phase1_supplementary_questions": [],
    "phase2_modification_hints": []
  }
}
```

- phase-agent は SKILL.md の prompt パラメータでプロファイルパスを受け取り Read ツールで読み込み
- 初期は代表2ジャンル（タロット・四柱推命）のみ作成

#### (2) リッチ JSON 要約スキーマの定義

desire-architecture 生成後に抽出する JSON 要約の必須フィールド:

| フィールド | 内容 | 理論上の根拠 |
|-----------|------|-------------|
| `target_profile` | ターゲット情報 | Phase 1 出力 |
| `genre_context` | ジャンル文脈 | プロファイル参照 |
| `desire_analysis` | 表層/深層/活性化タイプ配列 | 欲求4類型 |
| `wall_analysis` | `dependency_order` 配列 + 各壁の厚さ/層/解体戦略 | 壁4タイプ + 依存順序 |
| `cross_function_techniques` | 両軸に効く手法のリスト | 理論第5章 |
| `design_warnings` | 注意テキスト配列 | 「覚悟の教育」等 |
| `differentiation_axes` | 各軸の主欲求/支援欲求/強度根拠 | テンプレート Section 4 |
| `phase_desire_map` | 集客/体験/リピート/導線のマトリクス | 理論フェーズ別表 |

JSON Schema として `references/` に配置しバリデーション強制。

#### (3) Phase 1 バイアス補正機構

phase1-agent プロンプトに3段補正を組込:

1. **Q3 回答後**: ジャンルプロファイルの典型壁パターン提示 → 照合質問
2. **深掘り**: 具体的顧客エピソード（「最近お断りされたお客様のケースを教えてください」等）
3. **確認提示**: マッピング結果を平易な言葉で提示し確認

内部マッピングは**ハイブリッド方式**（プロファイルの `wall_thickness_guide` でルール粗分類 → LLM プロンプト内推論で微調整）。補正質問は Progressive Disclosure とし不確実性が高い場合のみ発動。

#### (4) カスケード検知の3段構成

1. **静的依存グラフ定義**: 5ドキュメントの依存順序（ロック済み）をそのまま使用
2. **ファイルハッシュ変更検知**: phase-state.json の `completed_documents[]` にタイムスタンプとハッシュを管理
3. **ユーザー確認**: 上流変更検知時に平易な確認メッセージ → 明示的承認後にのみ下流再生成

Phase 遷移制御は2層分離（内部: SKILL.md 排他書込のシリアライズパターン、表示: ユーザー向け進捗メッセージ）。

#### (5) 活性化強度スコアリングルール

理論第5章クロスファンクション表を基準:

| 強度 | 定義 |
|------|------|
| **高** | クロスファンクション表に登場 かつ 3フェーズ以上カバー |
| **中** | 片軸のみ だが 2フェーズ以上カバー |
| **低** | 片軸 かつ 1フェーズのみ |

`references/scoring-rules.md` に文書化し、テンプレートと phase-agent プロンプトの両方から参照。

### フォールバック戦略

以下のいずれかが発生した場合、簡素化構成に退避:

| トリガー | 退避内容 |
|---------|---------|
| JSON プロファイル読込でコンテキスト肥大化 → 応答品質劣化 | ジャンルロジックをプロンプト内条件分岐に変更、genres/ 廃止 |
| リッチ JSON 要約の抽出成功率 80% 未満（3回調整後も） | 簡易版5フィールドに縮退 + 本文直接参照で補完 |
| 初期2ジャンルのプロファイル作成に 40h 超 → 2週間以上遅延 | 上記の簡素化構成に退避 |

### 中止条件

占い・スピ系事業者 3名以上のプロトタイプテストで **Phase 1 完了率 50% 未満** または **Phase 2 修正操作成功率 30% 未満** の場合、SKILL.md ベース CLI 実装の前提自体が崩壊。Web UI 等の非 CLI 実装形態への根本転換が必要。

> 中止時の機会損失: 理論・テンプレート・倫理ルールはフォーマット非依存のため将来 Web UI 等で再利用可能。完全損失ではなく実装形態変更コスト（推定 $5,000〜$15,000）が発生。
