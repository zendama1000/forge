## 1. エグゼクティブサマリー

6視点の横断分析から、以下の **5つの設計判断軸** が浮上した。全視点に共通する最重要原則は「**決定論的メカニズムとLLM柔軟性の適切な分離**」であり、Phase遷移・テンプレート変数展開・整合性検証・倫理ゲートのいずれにおいても、LLMの自然言語指示のみに依存する設計は信頼性不足である。

| # | 設計判断軸 | 推奨方針 | 信頼度 |
|---|-----------|---------|--------|
| 1 | サブエージェント分割 | Phase別3体 + ジャンル別委譲のハイブリッド | 高 |
| 2 | Phase 1 ヒアリング | 3〜5項目の日常語質問 + Progressive Disclosure | 高 |
| 3 | Phase遷移制御 | JSONベース状態管理 + ファイル存在チェックの決定論的ゲート | 高 |
| 4 | テンプレート間整合性 | 自由形式生成→JSON要約抽出→下流注入のパイプライン | 中 |
| 5 | 倫理ゲート | 静的パターン(第1層) + LLMジャッジ(第2層)の二段構成 | 高 |

| 項目 | 範囲 |
|------|------|
| **深さ** | 選択肢の列挙→トレードオフ分析→推奨案と根拠の3段階まで。コード設計には踏み込まない |

## 3. 視点別調査結果

### 3.1 技術的実現性（Technical）

#### Phase状態管理
- SKILL.mdにネイティブ状態管理機構は **存在しない**
- 推奨: 3層構造で永続化
  1. **JSONファイル** — Phase番号・生成済みドキュメント・ユーザー選択値
  2. **agent-memoryディレクトリ** — サブエージェント間の永続知識
  3. **CLAUDE.md / MEMORY.md** — セッション横断の文脈

#### ファイル参照パス
- Readツールはプロジェクトルート基準で相対パスを解決
- サブエージェントはfreshスタートのため、プロンプト内にパスを **明示的に記述** する必要あり
- 既知バグ: サブエージェントのskills解決がグローバルパスのみ参照（Issue #10061）

#### カスケード検知
- Bash + `git diff` でSKILL.md内から **技術的に実行可能**
- ただしBashツールがallowed-toolsに含まれている必要あり
- `!`バックティック構文にセキュリティ上の既知問題あり（Issue #14315）

#### テンプレート変数展開
- ネイティブサポートは `$ARGUMENTS` / `$0` / `$1` のみ
- `{{desire_type}}` 形式のカスタム変数はネイティブ展開 **不可**
- 推奨: Writeツール経由でエージェントに読取→置換→書込を実行させる

#### ジャンル別テンプレート構成
- 推奨: **ハイブリッド**（共通フローをSKILL.mdに持ち、ジャンル固有処理を専用サブエージェントに委譲）
- 単一テンプレート＋条件分岐は指示数100〜150件で信頼性低下のリスク

### 3.2 コスト・リソース（Cost）

#### サブエージェント分割のコスト比較

| 方式 | エージェント数 | 開発コスト | テストコスト | 保守コスト |
|------|-------------|-----------|------------|-----------|
| **(A) Phase別** | 3 | 低 | 低 | 低 |
| **(B) ドキュメント別** | 6 | 高（実質2倍+） | 高（表面積2倍） | 高 |
| **(C) 機能別** | 3 | 低 | 低 | 低 |

- 研究では4エージェント超でパフォーマンス飽和が発生
- 逐次処理タスクでは多エージェント構成が39〜70%性能劣化する事例あり

#### ヒアリング項目のバランス点
- UX研究: 質問数は **7項目以下** が完了率維持の上限
- 最適解: **3〜5項目の必須ヒアリング + 段階的開示（Progressive Disclosure）**
- 質問の「数」より「具体性・構造性」が出力品質に寄与

#### テンプレート整合性検証
- 手動レビューはスケーラビリティに根本的制約
- 自動チェック（旧consistency-scorer.ts概念流用）の初期実装: **1〜2日**
- 手動で残すべき領域: ニュアンスのある意味整合性判断のみ

#### トークンコスト
- 6ドキュメント全載せ vs 選択的参照: **RAGで75%削減・精度95%維持**
- プロンプトキャッシング活用で **最大90%コスト削減**
- 「コンテキスト中間消失」問題への対策として選択的参照が有効

#### 旧コード抽出
- 対象: 7禁止パターン + 5プラットフォーム仕様 + 3法令区分（計15件）
- **手動作業で十分**（1〜2営業日）、自動スクリプト開発は過剰投資

### 3.3 リスク・失敗モード（Risk）

#### 高リスク（要即時対策）

| リスク | 信頼度 | 対策 |
|--------|--------|------|
| **Phase 1 生成品質**: 2入力のみでは汎用的・抽象的な出力になる | 高 | ロールアサイン + Few-shot例示 + 自己評価ループ + 出力フォーマット定義 |
| **倫理ゲート漏れ**: 静的7パターンでは変形表現を捕捉不可 | 高 | セマンティック分類器 + LLMジャッジの二重チェック |

#### 中〜高リスク（設計段階で対処）

| リスク | 信頼度 | 対策 |
|--------|--------|------|
| **テンプレート間不整合**: エラーカスケード伝播・合意慣性 | 中 | グローバルバリデーションエージェント + 依存グラフ管理 |
| **Phase遷移誤動作**: コンテキスト劣化・社会的手がかりバイアス | 中 | 決定論的コードによるPhase状態管理 |
| **カスケード未検知**: 上流変更の下流未伝播 | 中 | 明示的依存グラフ + 変更タイムスタンプ比較 |

#### 中リスク（許容可能だが監視必要）

| リスク | 信頼度 | 備考 |
|--------|--------|------|
| **ドメイン適用範囲**: 物販系スピリチュアルへのフレームワーク不適合 | 中 | 補償的コントロール心理という特殊動機の存在。要実証テスト |

### 3.4 代替案比較（Alternatives）

#### サブエージェント分割

```
推奨: (A) Phase別ベース + ジャンル別サブエージェント委譲のハイブリッド

(A) Phase別   — 責任範囲明確、デバッグ容易。クロスフェーズ再利用が困難
(B) ドキュメント別 — 1:1対応が直感的。一貫性担保が課題、エージェント数増大
(C) 機能別    — 再利用性最高。インターフェース設計コスト高、責任境界不明瞭
```

#### Phase 1 ヒアリング方式

```
推奨: (A)+(B) ハイブリッド — 最小入力で開始し、Progressive Disclosureで詳細化

(A) 最小入力→全自動 — 摩擦最小だがAI誤解リスク高
(B) 段階的ヒアリング — 認知負荷分散、新規ユーザーに有効
(C) テンプレート選択式 — 決定麻痺防止だが柔軟性低
```

#### カスケード検知

```
推奨: (C) hookベース + LLMによる差分解釈

(A) SKILL.md指示のみ  — 最安だが信頼性低
(B) 専用エージェント   — 高信頼だが高コスト
(C) pre/post hook     — 決定論的で確実。複雑依存には限界あり
```

#### テンプレート間参照

```
推奨: (B)+(C) ハイブリッド — JSON要約 + 必要時の選択的全文参照

(A) 明示的セクション参照  — トレーサビリティ高、変更に脆弱
(B) 中間JSONモデル       — 型付きデータ伝達、ただしLLM推論能力10-15%低下リスク
(C) 全文コンテキストチェーン — ナラティブ一貫性最高、トークン消費大
```

#### 差別化軸提示

```
推奨: (B) 推奨1案 + 代替2案の優先度付き提示

(A) 3案並列   — 探索フェーズ向き、最終提示には不向き
(B) 推奨+代替  — 認知負荷低減、「驚き」排除。UX研究で最も支持
(C) 対話的探索 — 最も柔軟だが最も遅い（30-60秒の滞留問題）
```

### 3.5 ユーザー対話体験設計（Interaction Design）

#### 専門概念の日常語翻訳
- **プログレッシブ・ディスクロージャー + プレイン・ランゲージ原則** が有効
- 専門用語は一切表出させず、収集した自然言語回答をバックエンドでドメインモデルに変換
- 質問例:
  - 「あなたのお客様は何に困っていますか？」 → 欲求類型マッピング
  - 「なぜ今まで解決できなかったと思いますか？」 → 壁の特定
  - 「競合と比べてあなたの強みは？」 → 差別化軸マッピング

#### 差別化軸操作の抽象化
- **抽象化レイヤーパターン** で内部操作を日常的質問に隠蔽
- ただし会話型UIだけでは複雑操作が遅くユーザーを疲弊させるリスクあり
- 推奨: テキスト質問 + 視覚UIコントロール（選択肢・スライダー）のハイブリッド

#### Phase遷移のUX
- 購買設計ドキュメントは重要な業務文書 → **明示的確認 + 生成内容プレビュー** が有利
- 自動遷移は可逆的操作のみ。Phase遷移は半可逆（戻れるが再入力コスト大）

#### 進捗可視化
- **ステッパーUI（Step X of Y）** が最も実証豊富
- 3状態の視覚的区別: 完了済み（チェック）/ 進行中（強調）/ 未実施（グレー）
- 各ステップにドキュメント名ラベルを添付

### 3.6 理論的忠実性・ドメイン整合（Domain Fidelity）

#### 3つの構造的忠実性リスク

**リスク1: 表層/深層欲求の平坦化**
- LLMは明示的スキーマ制約なしに欲求の層構造を単一次元に圧縮する傾向
- 壁①の3層構造のような内部区別が失われやすい
- 対策: 欲求類型ごとの **明示的スキーマ定義**（表層/深層を必須フィールド化）

**リスク2: 壁の解体順序性の消失** ← 最も具体的かつ修正可能
- 一般消費者心理研究でも購買障壁の依存順序は支持されている
  - 日本: 知らない→興味ない→比較しない→納得しない→行動しない（JCPO「5つの壁」）
  - 欧米: cynicism→skepticism→procrastination→price
- 現行ドラフトの並列記述はこの **非可換性を破壊** している
- 対策: **Step 1→2→3→4の依存グラフ構造** で記述

**リスク3: 差別化軸と欲求類型の紐付け消失**
- 欲求活性化マッピングなしの自動生成は理論的根拠のない軸を生成するリスク
- 対策: 各差別化軸に「活性化する欲求類型」「活性化強度（高/中/低）」フィールドを追加

#### ジャンル別壁の厚さ分布（信頼度: 低）

| ジャンル | 認識障壁（壁①相当） | 信頼障壁 | 備考 |
|---------|-------------------|---------|------|
| 占星術・タロット | 薄い（認知度高） | 中（科学的根拠への懐疑） | 娯楽性が購買障壁を相殺 |
| ヒーリング | 厚い（作用機序不透明） | 高 | 初回接触の閾値が高い |
| 物販系スピ | 中（商品は認識しやすい） | 品質証明壁が主要 | 信頼壁より品質壁が支配的 |

> **注意**: 壁の厚さ分布を理論から演繹するロジックは公開情報で確認できず、学術研究も極めて限定的。

## 4. 視点間の矛盾と解決

### 4.1 サブエージェント分割のコスト評価（Cost vs Alternatives）

| 視点 | 主張 |
|------|------|
| Cost | Phase別(3体)と機能別(3体)のコスト差は「ほぼ等しい」 |
| Alternatives | 機能別はインターフェース設計コストが追加で発生し、責任境界が不明瞭 |

**解決**: Phase別をベースとし、Phase内で必要に応じて機能分割するハイブリッド構成を採用。

### 4.2 カスケード検知の信頼性（Technical vs Risk）

| 視点 | 主張 |
|------|------|
| Technical | Bash + git diff でSKILL.md内から実行可能（高信頼度） |
| Risk | 自然言語指示のみでは長期的に中〜高リスク（コンテキスト劣化） |

**解決**: git diff実行はBashで行い、実行タイミング・条件判断は **決定論的スクリプト** で制御。LLMには差分の「解釈」のみを委ねる。

### 4.3 JSON構造化の必要性と品質低下（Alternatives vs Domain Fidelity）

| 視点 | 主張 |
|------|------|
| Alternatives | JSON強制でLLM推論能力10-15%低下のリスク |
| Domain Fidelity | 欲求類型の明示的スキーマ定義（JSON）が理論忠実度保存に不可欠 |

**解決**: 「推論フェーズは自由形式→後処理でJSON変換」の分離パターン。生成後に重要フィールドのみJSON要約として抽出・検証。

### 4.4 専門概念の翻訳と忠実性（Interaction Design vs Domain Fidelity）

| 視点 | 主張 |
|------|------|
| Interaction Design | 専門用語を一切ユーザーに見せない（日常語翻訳） |
| Domain Fidelity | 壁①の3層構造や欲求類型の区別をスキーマで厳密に保存すべき |

**解決**: **表裏分離アーキテクチャ** — ユーザー対話層は日常語に完全翻訳、システム内部層では理論の正式用語・構造を厳密に保持。

### 4.5 倫理ゲートの投資水準（Cost vs Risk）

| 視点 | 主張 |
|------|------|
| Cost | 15件の旧コード抽出は手動で十分（1-2日）、自動スクリプトは過剰投資 |
| Risk | 静的7パターンのみでは高リスク、セマンティック検知が必要 |

**解決**: 両者は異なるレイヤーを論じている。旧パターンは **手動抽出**（Cost効率）、運用時は **静的マッチ + LLMジャッジの二段構成**（Risk対策）。

## 5. 推奨事項

### 5.1 Primary推奨: 5つの構造的補強を伴う実装

#### (1) サブエージェント分割
- **Phase別3体** + **ジャンル別サブエージェント委譲** のハイブリッド
- 共通フロー制御はSKILL.mdに集約
- ジャンル固有ロジックは `.claude/agents/genres/` 以下に分離
- 同時起動は常に **3体以下** に制限

#### (2) Phase 1 ヒアリング拡張
- 2入力 → **3〜5項目の日常語必須質問 + Progressive Disclosure**
- 質問設計:
  1. 占術ジャンル
  2. ターゲット属性
  3. 「お客様が申し込みを迷う最大の理由は？」 → 壁の特定
  4. 「競合と比べてあなたの強みは？」 → 差別化軸
- 専門用語は一切表出させず、内部マッピングで処理

#### (3) Phase遷移の決定論的制御
- `state/phase-state.json` に状態を管理:
  ```json
  {
    "current_phase": 1,
    "completed_documents": [],
    "user_selections": {}
  }
  ```
- Phase遷移条件: ファイル存在チェック（例: `output/desire-architecture.md` exists → Phase 2進行可）
- SKILL.md指示 + サブエージェント起動時のPhase状態注入で **多層防御**

#### (4) テンプレート構造改修
- 壁の解体戦略: 並列記述 → **Step 1→2→3→4の依存グラフ構造**
- 差別化軸: 各案に「活性化する欲求類型」「活性化強度」フィールド追加
- テンプレート間参照: **自由形式生成→JSON要約抽出→下流注入** のパイプライン

#### (5) 倫理ゲート二段構成
- **第1層**: 旧7パターンの静的マッチ（正規表現）
- **第2層**: LLMジャッジによるセマンティック評価（景品表示法・特商法・倫理基準）
- 実行ポイント: Phase 1生成完了時 + Phase 2修正完了時

### 5.2 Fallback: 最小構成

**切り替えトリガー**:
1. Primary構成の実装工数が見積もり2倍超
2. サブエージェント間のファイル参照パス問題（Issue #10061等）が解決不能
3. Phase状態JSONの読み書きがコンパクションと競合し不安定

**内容**:
- サブエージェント: 2体（generator + validator）に縮小
- Phase遷移: SKILL.md + ファイル存在チェックの簡易構成
- ジャンル分岐: 単一テンプレート内条件分岐
- 倫理ゲート: 静的7パターンのみ（LLMジャッジは後続イテレーション）

### 5.3 中止条件

以下の **2条件が重複** した場合、プロジェクト中止を推奨:
1. 4欲求×4壁フレームワークが主要3ジャンル中2つ以上で根本的に不適合と実証
2. Claude Codeスキルのセッション間状態永続化が実用に耐えない
