## 3. 視点別調査結果

### 3.1 技術的実現性 (confidence: HIGH)

#### コンテキスト上限とサンプル数

- 公称コンテキストウィンドウ: 標準 200K、有料プラン 1M トークン
- Claude Code 内部バッファ予約: 約 33K トークン（実質 167K 利用可能）
- 日本語ツイート 1 件 = 約 90-140 トークン
- **実用的天井はコンテキストロット（精度劣化）で決まる → 20-50 件が妥当**

#### Claude Code 呼び出し方式

| 方式 | 安定性 | 備考 |
|------|--------|------|
| `-p` フラグ（非対話） | **最も安定** | `--output-format json` でパース可能 |
| stdin パイプ | 要注意 | 7000 文字超で空出力バグ（Issue #7263） |
| Agent SDK | 要注意 | Windows+Python 3.12 で RuntimeError あり |

- **推奨**: `-p` フラグ + CLI 引数渡し。引数長超過時のみ一時ファイル経由にフォールバック
- v2.1.81 の `--bare` フラグでスクリプト実行に最適化された軽量モードが利用可能

#### ツール実装言語

**Node.js CLI が最適**。シェルスクリプトと比較して：
- 成熟した CLI フレームワーク（Commander.js / yargs / oclif）
- 設定ファイル（JSON/YAML）の堅牢な読み書き
- クロスプラットフォーム対応
- テスト容易性

#### 280 文字制限の遵守

| 手法 | 遵守率 | 備考 |
|------|--------|------|
| プロンプト単純指示のみ | **30% 未満** | 不十分 |
| カウントダウンプロンプト | **95% 超**（GPT-4.1 実験） | Claude では未検証 |
| ドラフト → 修正ループ | 高（未定量） | 実績あり |
| プログラム的バリデーション + リトライ | 高 | 最大 3 回推奨 |

**推奨**: カウントダウンプロンプト + プログラム的バリデーション + 最大 3 回リトライの 3 層防御

### 3.3 リスク・失敗モード (confidence: HIGH)

#### リスクマトリクス

| リスク | 深刻度 | 発生確率 | 対策 |
|--------|--------|----------|------|
| サンプル丸写し・微修正 | 中 | 中-高 | コサイン類似度チェック + negative prompting + temperature 調整 |
| 文体パラメータの抽象性 | 中 | 高 | 具体的ルール列挙（「体言止め」「絵文字最大 2 個」等） |

| サンプルデータ品質劣化 | **高** | 中 | quality_score フィールド + 手動レビュー |

#### Few-shot 崩壊の定量データ

低品質サンプル混入時の性能低下実績：
- Gemma 7B: 77.9% → 39.9%
- LLaMA-2 70B: 68.6% → 21.0%

→ **品質フィルタの実装が不可欠**

#### Claude Code 依存リスクの緩和

1. 呼び出しを単一ラッパー関数に集約
2. 呼び出し方式（`-p`、stdin、一時ファイル）を設定で切替可能
3. 出力パース処理もラッパー内に封じ込め
4. 将来の API 直接利用への移行もラッパー差替えのみで対応

### 3.5 対話 UX・ワークフロー設計 (confidence: HIGH)

#### 対話フロー設計原則

**二重モード提供が標準解**:
- **ウィザードモード**（初回・学習用）: ステップごとにプロンプト入力を促す
- **フラグモード**（パワーユーザー用）: `--genre tech --count 3 --yes` で一括実行

#### 候補提示 UX

```
--- ツイート候補 ---

1. AIエージェントの進化が止まらない。
   Claude Codeで実装自動化したら、
   1日の作業が2時間で終わった件について。

2. 最近のLLM、コード書くだけじゃなくて
   設計判断まで提案してくるのすごい。
   人間の仕事が「レビュー」に変わりつつある。

3. 「まずMVPを作れ」の意味が変わった。
   AIと2時間でプロトタイプできる時代、
   MVPはもはや最小限じゃなくて最速の検証。

[1/2/3/e(edit)/r(regenerate)] >
```

- **3 件一括表示**が業界主流
- フィードバックは「採用 / 再生成 / 部分微調整」の 3 択が最小セット

#### ショートカット設計

| フラグ | 機能 |
|--------|------|
| `--same` / `-s` | 前回設定をそのまま引き継ぎ再生成 |
| `--preset <name>` | 名前付きプリセット呼出 |
| `--count N` | 候補数指定 |
| `--yes` | 全デフォルト採用（対話スキップ） |

設定継承の 3 層: グローバル設定 → ローカル設定 → CLI フラグ

### 3.6 データ管理・ジャンル分離アーキテクチャ (confidence: HIGH)

#### 推奨ディレクトリ構成

```
data/
├── genres/
│   ├── tech/
│   │   ├── samples/
│   │   │   ├── sample-001.yaml
│   │   │   ├── sample-002.yaml
│   │   │   └── ...
│   │   ├── style.json
│   │   ├── references/
│   │   └── history.jsonl
│   ├── lifestyle/
│   │   ├── samples/
│   │   ├── style.json
│   │   ├── references/
│   │   └── history.jsonl
│   └── ...
└── shared/
    └── (共有リソース)
```

- ジャンルを最上位単位としたネスト構造（マルチテナント設計の知見と合致）
- 浅い階層（2-3 段）で認知負荷を最小化
- ジャンル 5 以上でネスト構成の優位性が顕著に

#### スタイル設定スキーマ

```json
{
  "schema_version": "1.0.0",
  "tone": "casual",
  "constraints": ["1ツイート最大2個の絵文字", "ハッシュタグは末尾に1-2個"],
  "emoji_policy": { "frequency": 0.3, "max_per_tweet": 2 },
  "sentence_endings": ["体言止め", "〜だよね", "〜かも"],
  "hashtag_rules": { "enabled": true, "max_count": 2 }
}
```

- 新規パラメータは全て **optional + デフォルト値付き**で追加（後方互換性）
- パラメータの削除・リネームは**禁止**
- 未知フィールドは無視（tolerant reader pattern）

#### 履歴管理

| 段階 | 方式 | 用途 |
|------|------|------|
| **MVP** | **JSONL 追記のみ** | git-friendly、マージコンフリクトなし |
| 拡張期 | JSONL + SQLite ハイブリッド | 高速重複チェック、ジャンルフィルタ |

- 重複防止: SHA-256 コンテンツハッシュ（完全一致）
- 各エントリ: `{timestamp, genre_id, content_hash, generated_text, quality_score}`
- SQLite 導入は履歴 1000 件超の段階で検討

## 4. 統合分析: 矛盾と解決策

### 4.1 矛盾 1: コストモデルの不一致

| | Cost 視点の前提 | ロック決定 |
|---|---|---|
| 課金モデル | API 直接課金（per-token） | Claude Code サブスク定額（$20-200/月） |
| キャッシュ | プロンプトキャッシュでコスト圧縮 | CLI 経由では制御不能 |

**解決**: 定額制では per-tweet コスト最適化は不要。利用頻度（月何ツイート生成するか）が ROI の軸になる。

### 4.2 矛盾 2: 二重対話レイヤー

Node.js CLI（Inquirer）と Claude Code の 2 つの対話レイヤーが重なる問題。

**解決**: 責務を明確に分離。
- **Node.js CLI**: ジャンル選択・設定確認・候補選択（構造化入力）
- **Claude Code `-p`**: テキスト生成のみ（非対話的に実行）

ユーザー対話は Node.js CLI に一元化し、Claude Code は「生成エンジン」として使用。

### 4.3 矛盾 3: 履歴管理の過剰設計

Data Architecture 視点は JSONL + SQLite ハイブリッドを推奨するが、MVP 工数 8-16 時間の目標と矛盾。

**解決**: 段階的採用。MVP は JSONL のみ（O(N) 全件走査、N<1000 なら問題なし）。SQLite は 1000 件超で導入。

### 4.4 矛盾 4: 全サンプル埋込 vs Few-shot 崩壊

全サンプル埋込は低品質サンプルも無差別に含むため、崩壊リスクが最大化。

**解決**: 保存時に品質フィルタを実装。
1. 登録時に URL 過多・極端短文を warning 表示
2. `quality_score`（手動 5 段階）で score >= 3 のみ埋込
3. 20 件超はランダムサンプリングまたは類似度選択に切替

## 5. 推奨アクション

### 5.1 Primary: Node.js CLI ラッパー型

**Commander.js + Inquirer.js で対話フロー、Claude Code `-p` を生成エンジンとして使用**

| 技術選定 | 決定 |
|---------|------|
| サンプル保存 | YAML |
| 文体設定 | JSON（schema_version 付き） |
| ディレクトリ | `data/genres/{name}/samples/*.yaml, style.json, references/, history.jsonl` |
| 候補提示 | 3 件一括・番号付きリスト |
| 280 文字制限 | カウントダウンプロンプト + バリデーション + 最大 3 回リトライ |
| サンプル数 | 10-20 件/ジャンル |
| Claude Code 呼出 | 単一ラッパー関数、`-p` + CLI 引数渡し |
| 履歴管理 | JSONL + SHA-256 重複検出 |

#### 残留リスク

1. Claude Code CLI の破壊的変更（年 176 回）→ ラッパー層で吸収するが運用コストは継続発生
2. `-p` モードの安定性が未実測（stdinバグが CLI 引数でも発生する可能性）
3. 日本語 280 文字制限のカウントダウンプロンプト有効性が Claude 系モデルで未検証
4. サンプル品質の暗黙的劣化（手動収集のため機械的検出が困難）
5. SNS 文体は LLM が最も模倣しにくいジャンル（2025 年研究）

### 5.2 Fallback: Claude Code ネイティブ型

**Node.js CLI 不使用。CLAUDE.md + スラッシュコマンドで構成。**

- MVP 工数: **2-4 時間**（Primary の 1/4）
- 対話レイヤーが 1 つに統一
- ジャンルデータのディレクトリ構造は Primary と同一 → 将来の移行も容易
- 280 文字バリデーション・リトライはプロンプト指示のみ（遵守率低下を許容）

**切替トリガー**:
1. Claude Code `-p` モードのバグが CLI 引数渡しでも回避不能
2. サブプロセス起動の成功率が 95% 未満
3. MVP 開発工数が見積もりの 2 倍（32h 超）に到達

### 5.3 撤退条件

1. Claude Code CLI が毎月動作不能になりラッパーで吸収不可
2. 日本語文体模倣品質が実用に耐えない（生成の 50% 以上が手動書き直し必要）
3. Claude Code サブスクの廃止・大幅値上げで既存 SaaS（$8/月～）より高コストに

**撤退時の機会損失**: ジャンル別文体学習による 3-5 倍の時間短縮、x-auto-agent 連携パイプラインへの発展、ローカル実行 + プライバシー保護という差別化価値
