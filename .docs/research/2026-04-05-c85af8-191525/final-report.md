## 1. 調査概要

本調査は、**ジャンル別に収集したサンプルツイートを素材として、Claude Code の対話モードでオリジナルツイートを生成する CLI ツール**の設計方針と技術的実現性を、6つの視点から多角的に検証したものである。

### 調査視点

| # | 視点 ID | フォーカス | 確度 |
|---|---------|-----------|------|
| 1 | `technical` | 技術的実現性 | 高 |
| 2 | `cost` | コスト・リソース | 中〜高 |
| 3 | `risk` | リスク・失敗モード | 高 |
| 4 | `alternatives` | 代替案・競合 | 中〜高 |
| 5 | `ux_workflow` | ユーザー体験・操作フロー | 高 |
| 6 | `data_architecture` | データモデル・拡張性 | 高 |

### 前提として露出された仮定

調査の冒頭で以下の暗黙的仮定が特定された:

- Claude Code 対話モードでツイート生成に十分なプロンプト制御が可能という前提（実際は CLAUDE.md / カスタムコマンドでの制御に限定）
- 手動コピペ収集で十分な量を集められるという前提（1ジャンル何件が最低ラインか未定義）
- 1ツイート = 1ジャンルの排他的所属という暗黙前提（クロスジャンルの存在）
- 生成ツイートが日本語であるという暗黙前提（多言語対応の要否未定義）
- 文体設定を宣言的に定義すれば LLM が忠実に従うという前提（プロンプトエンジニアリングの品質に強く依存）

## 2. 調査の核心的問い

1. ジャンル分離を実現するディレクトリ構成・データモデルの最適設計
2. サンプルツイートの保存形式（JSON / YAML / Markdown）の選定
3. トーン学習と参照コンテンツ融合のプロンプトパイプライン構成
4. Claude Code 対話モードの技術的制約への対処
5. 生成ツイートのオリジナリティ担保（パクリ防止）
6. x-auto-agent 連携を見据えたインターフェース境界
7. ジャンル設定ファイルの粒度と表現力のバランス
8. 生成フロー全体の最小構成と拡張ポイント

## 3. 視点別調査結果

### 3.1 技術的実現性（technical）

**結論: 実現可能。ただし出力形式の強制制御不可・セッション間状態管理に設計上の考慮が必要。**

| 調査項目 | 結果 | 確度 |
|---------|------|------|
| コンテキストウィンドウ | 標準 200K tokens（Pro/Max で 1M）。ツイート 5件で 750-1400 tokens と低コスト | 高 |
| カスタムスキル | `.claude/skills/` に Markdown で定義。`allowed-tools`、`context: fork`、`model` 指定が可能 | 高 |
| 出力形式制御 | ネイティブ強制は不可。プロンプト指示でのみ制御 | 高 |
| 状態永続化 | CLAUDE.md または外部ファイル（JSONL 等）への書き出しが必要 | 高 |
| few-shot 最適件数 | 2-5件がスイートスポット。5件超は過学習リスク | 高 |
| 文体指示方法 | 宣言的指示 + few-shot 例示の併用が最も忠実度が高い | 中 |
| 類似度チェック | all-MiniLM-L6-v2（23M パラメータ、約 80MB）が軽量・高品質で最適 | 高 |
| CLI フレームワーク | Commander.js + Inquirer.js が実績・型安全性で最適（ただし Claude Code スキルでは不要） | 高 |

### 3.2 コスト・リソース（cost）

**結論: MVP 0.5-1週間、月次ランニングコスト $5-25。ジャンル追加は線形コスト。**

#### トークン消費・月次コスト見積もり（100生成/日）

| 構成 | Sonnet 4.6 | Haiku 4.5 |
|------|-----------|-----------|
| 最小（5サンプル + 500字参照 + 1候補） | ~$0.004/回 | ~$0.0013/回 |
| ミドル（10サンプル + 1000字参照 + 3候補） | ~$0.008/回 | ~$0.0026/回 |
| 最大（20サンプル + 2000字参照 + 5候補） | ~$0.013/回 | ~$0.0044/回 |
| **月次コスト（ミドル構成）** | **~$23/月** | **~$8/月** |

- プロンプトキャッシュ活用で入力コスト 30-50% 削減可能
- 日本語はトークン消費が英語比 1.5-2 倍になる可能性あり（未実測）

#### 開発工数

| スコープ | 見積もり |
|---------|---------|
| MVP（ジャンル管理 + 生成フロー） | 3-7日（0.5-1週間） |
| 完全版（類似度チェック + 履歴 + 連携 IF） | +8-15日（トータル 2-3週間） |

#### サンプル収集負荷

- 1ジャンルあたり推奨 **5-10件**
- 収集時間: **30-90分/ジャンル**（初回は 60-120分）
- ジャンル追加の限界コスト: **1.5-2.5時間/ジャンル**（線形）

### 3.3 リスク・失敗モード（risk）

**結論: 6つの調査問い全てに実証的裏付けのある失敗モードが存在。設計段階で対策可能だが、完全解決は困難なものもある。**

| リスク | 深刻度 | 確度 | 対策可能性 |
|-------|--------|------|-----------|
| 少数サンプルでの過学習的模倣 | 高 | 高 | サンプル多様性確保 + 類似度チェックで軽減可能 |
| LLM 均質化によるジャンル分離失敗 | 高 | 高 | 詳細な文体定義で軽減するが完全解決は困難 |
| Claude Code 破壊的変更 | 中 | 高（実例あり） | 抽象層設計で影響範囲を局所化 |
| 参照コンテンツ品質ばらつき（コンテキストロット） | 中 | 高 | コンテキスト先頭/末尾への配置で軽減 |
| x-auto-agent 連携のスキーマ非互換 | 中 | 中 | スキーマファースト設計で回避可能 |
| 著作権・ToS グレーゾーン | 低〜中 | 中 | 個人・少数件・非商用なら実務リスク低 |

#### Claude Code 破壊的変更の実例

- v2.1.87 → v2.1.88 でカスタムコマンドが検出されなくなる回帰が発生（GitHub Issue #41497）
- 週 2-3 回のリリース頻度で、ツールのリネーム（View → Read 等）、npm → ネイティブバイナリ移行など複数の破壊的変更歴あり

#### コンテキストロット（Context Rot）

- Chroma Research（2025年）が 18 の最先端モデル全てで入力トークン数増加に伴う出力品質低下を実証
- 中間に配置された参照コンテンツは 30% 超の精度低下を引き起こす（U字型カーブ）

### 3.4 代替案・競合（alternatives）

**結論: ローカルファースト + 精密スタイル模倣 + 人間-in-the-loop に明確な差別化余地あり。LangChain は不要。**

#### サンプル保存形式の比較

| 形式 | トークン効率 | 編集容易性 | バリデーション | LLM精度 |
|------|-----------|-----------|-------------|---------|
| **Markdown** | 最高（JSON 比 34-38% 削減） | コピペ最速 | 困難 | 54.3% |
| **YAML** | 中（Markdown 比 +10%） | コメント可・可読性高 | 容易 | 62.1% |
| **JSON** | 最低（verbose） | 編集しにくい | 最も容易 | 50.3% |

> **推奨**: 用途別 3 層構成 — サンプル=Markdown、設定=YAML、履歴/キュー=JSONL

#### プロンプト構築戦略の比較

| 戦略 | F1スコア | 実装難度 | 推奨用途 |
|------|---------|---------|---------|
| (A) 全件 few-shot | — | 低 | 少数サンプル時のみ |
| (B) ランダムサンプリング | 65.69% | 低 | MVP |
| **(C) 関連度ベース選択（TF-IDF）** | **71.43%** | **中** | **完全版** |
| (D) 文体ルールのみ | — | 最低 | 非推奨（精度最低） |

#### 既存ツールとの差別化

| 既存ツール | 特徴 | 弱点 |
|-----------|------|------|
| AutoTweeter（LangChain + GPT-4） | クラウド依存・自動投稿 | スタイルカスタマイズ弱い |
| Twitter Intern（LangChain + Web） | フルスタック Web | CLI ではない |
| openai-twitter-bot（Python/Tweepy） | シンプル | スタイル制御なし |

**本ツールの差別化軸**: (1) ローカルファースト、(2) サンプルベース精密スタイル模倣、(3) ジャンル完全分離、(4) 人間-in-the-loop（自動投稿ではなくレビュー後投稿）、(5) CLI のシンプルさ

### 3.5 ユーザー体験・操作フロー（ux_workflow）

**結論: マルチステップウィザード + 3-5件候補提示 + インタラクティブ/非インタラクティブ両立が核心。**

#### 推奨対話フロー

```
ジャンル選択 → 参照コンテンツ入力（任意） → 生成実行 → 3-5件の候補提示
→ 番号選択 or 再生成 → 編集（任意） → 履歴保存 + キュー追加（任意）
```

#### サンプル登録の 3 モード

1. **stdin/パイプ入力**: `echo '...' | tool add-sample`（自動化向け）
2. **ファイルインポート**: バッチ登録（JSONL/Markdown）
3. **インタラクティブ入力**: ガイドレール付き対話

#### 生成候補提示

- **3-5件**が認知負荷と多様性のバランス点（SocialBee、Jasper 等の事例）
- バリエーション差異（トーン・長さ・アングル）を持たせることが効果を高める
- 「再生成」「編集」「削除」の 3 アクションをセットで提供

#### ジャンル管理のコマンド体系

```
tool genre create     # ジャンル新規作成
tool genre list       # 一覧表示
tool genre add-sample # サンプル追加
tool genre config     # 設定編集
```

### 3.6 データモデル・拡張性設計（data_architecture）

**結論: ジャンル軸の垂直スライス構造 + JSONL アペンドオンリー + Producer-Consumer キューが最適。**

#### ディレクトリ構造

```
genres/
├── tech/
│   ├── samples.md        # サンプルツイート（Markdown）
│   ├── config.yaml       # 文体設定（YAML）
│   └── history.jsonl     # 生成履歴（JSONL）
├── lifestyle/
│   ├── samples.md
│   ├── config.yaml
│   └── history.jsonl
└── ...
queue.jsonl               # x-auto-agent 連携用出力キュー
```

- **垂直スライス構造**: 1ジャンルの全ファイルが 1 ディレクトリに局所化（凝集度が高い）
- 新ジャンル追加時に複数フォルダを横断する必要がない

#### サンプルツイートのスキーマ（最小構成）

| フィールド | 型 | 必須 | 説明 |
|-----------|-----|------|------|
| `id` | string | 必須 | 一意識別子 |
| `text` | string | 必須 | ツイート本文 |
| `collected_at` | ISO8601 | 必須 | 収集日時 |
| `source_url` | string\|null | 任意 | 元 URL |
| `tags` | string[] | 任意 | ユーザー定義タグ |
| `public_metrics` | object\|null | 任意 | エンゲージメント数 |

#### 生成履歴（JSONL）

- **全候補保存**を推奨（ハッシュベース重複防止、採用率分析、品質フィードバック）
- JSONL アペンドオンリーは JSON 全体再書込みの **9,000 倍高速**

#### x-auto-agent 連携キュー

Producer-Consumer パターンのファイルベース実装:

| フィールド | 型 | 説明 |
|-----------|-----|------|
| `id` | string | ジョブ ID |
| `genre` | string | ジャンル名 |
| `text` | string | ツイート本文 |
| `status` | enum | `pending` → `posted` \| `failed` |
| `created_at` | ISO8601 | 作成日時 |
| `scheduled_at` | ISO8601\|null | 予約投稿日時 |
| `posted_at` | ISO8601\|null | 投稿完了日時 |

## 4. 統合分析・矛盾解消

6 視点の横断分析で 5 つの矛盾が検出され、以下のように解消された。

### 矛盾 1: 最適サンプル数（technical vs cost vs risk）

| 視点 | 主張 |
|------|------|
| technical | 2-3件でゲイン横ばい、5件超は過学習リスク |
| cost | 5-10件が品質と負担のバランス点 |
| risk | 1-2件で過学習実証済み、多様性 > 件数 |

> **解消**: **3-5件**を推奨レンジとして採用。「最低 3 件、多様性確保を条件に 5 件まで」。完全版では TF-IDF による動的選択（プール 10 件超から最適 3-5 件を選択）で全視点が収束。

### 矛盾 2: サンプル保存形式（technical vs alternatives vs data_architecture）

> **解消**: 用途別に形式を分離。サンプル=**Markdown**、設定=**YAML**、履歴/キュー=**JSONL**。3 形式併用のファイル数増加は `genres/<name>/` 垂直構造で局所化されるため管理負荷は許容範囲。

### 矛盾 3: 実行モデル（technical/alternatives vs ロック決定）

技術視点が推奨する Commander.js + Inquirer.js の Node.js CLI と、ロック決定「Claude Code で対話的に生成（API 不使用）」の間に概念的ギャップ。

> **解消**: ロック決定の意図を「**Claude Code カスタムスキルとして実装し、Claude Code の対話 UI をそのまま活用**」と解釈。Commander.js/Inquirer.js は不要となり、実装複雑度が大幅に低下。将来の API 移行時に導入する設計余地は残す。

### 矛盾 4: ジャンル分離の実効性（risk vs technical）

- risk: LLM の構造的均質化傾向で文体分離が失敗する（高確度）
- technical: 宣言的指示 + few-shot 併用で最も忠実度が高い（楽観的）

> **解消**: risk の警告を設計制約として受け入れ、多層防御を設計に組込む: (1) 具体的語彙リスト・NGパターン・構文制約の必須化、(2) few-shot 例示、(3) 生成後のジャンル間類似度モニタリング（完全版）。**完全排除は不可能であることをユーザーに明示**。

### 矛盾 5: UX パターンの適用（ux_workflow vs ロック決定）

> **解消**: ux_workflow の知見を Claude Code スキルのプロンプト設計に翻訳。マルチステップウィザード → スキル内の段階的質問。候補提示 → 番号付きリスト出力 + ユーザー選択。ジャンル管理 → サブコマンド的スキル分割。

## 5. 推奨設計方針（GO 判定）

### 判定: **GO**

6 視点の高確度知見が収束し、技術的に実現可能、差別化余地も明確。

### 設計方針サマリー

| 設計要素 | 採用方針 |
|---------|---------|
| **実行モデル** | Claude Code カスタムスキル（`.claude/skills/`）。外部 CLI フレームワーク不使用 |
| **ディレクトリ構造** | `genres/<name>/` 垂直スライス |
| **データ形式** | サンプル=Markdown、設定=YAML、履歴/キュー=JSONL |
| **プロンプト戦略** | 宣言的スタイル指示（config.yaml） + few-shot 例示（samples.md から 3-5 件） |
| **文体定義ルール** | 語彙リスト・NGパターン・構文制約を必須。曖昧な形容詞のみの指示は禁止 |
| **類似度チェック** | MVP: SHA256 ハッシュ完全一致。完全版: all-MiniLM-L6-v2 埋め込み（閾値 0.85） |
| **x-auto-agent 連携** | JSONL ファイルキュー（Producer-Consumer）。現時点は書き出しのみ |
| **MVP 範囲** | 2 ジャンル + 生成フロー + 基本履歴保存。開発 3-5 日 |

### スキル分割

| スキル名 | 機能 |
|---------|------|
| `generate` | ジャンル選択 → ツイート生成 → 候補提示 → 保存 |
| `add-sample` | サンプルツイート登録 |
| `list-genres` | ジャンル一覧表示 |
| `manage-genre` | ジャンル設定の編集 |

### フォールバック計画

**トリガー条件**: Claude Code スキルの破壊的変更が 2 回以上発生 / 出力形式制御の精度が 50% 未満 / コンパクション頻発

**代替構成**: `claude -p` フラグを活用したシェルスクリプト + Commander.js CLI のハイブリッド構成に移行。Forge Harness で実証済みの安定した呼び出し方法。

### 中止条件

以下の 3 条件のうち 2 つ以上が該当する場合:
1. 日本語ツイートでの few-shot 文体再現が実用品質に達しない（80% 以上がジャンル間で区別不能）
2. Claude Code の仕様変更で再実装コストが MVP 開発工数を超過
3. 著作権問題で法的助言が「使用不可」と明確に判断

## 6. 実装基準（Implementation Criteria）

### フェーズ構成

| **Polish** | エラーハンドリング・並行安全性・永続性の堅牢化 | L2 x2 |

### Layer 1 基準（ユニットテスト / API テスト）

| ID | 対象 | テスト種別 |
|----|------|-----------|

| L1-007 | 出力キュー JSONL の CRUD | unit_test |

### Layer 2 基準（E2E / 統合テスト）

| ID | 対象 |
|----|------|

## 7. リスクと対策

### 高優先度リスク

| # | リスク | 対策 |
|---|-------|------|
| 1 | **少数サンプルでの過学習的模倣** | 最低 3 件の多様なサンプルを確保。SHA256 重複検出（MVP）+ 意味的類似度（完全版） |
| 2 | **LLM 均質化によるジャンル分離失敗** | 具体的語彙リスト・NGパターン・構文制約を必須化。曖昧な形容詞のみの指示を禁止 |
| 3 | **Claude Code 破壊的変更** | スキルの動作確認を定期実施。フォールバック（`claude -p` モード）を準備 |

### 中優先度リスク

| # | リスク | 対策 |
|---|-------|------|
| 4 | **コンテキストロット** | 参照コンテンツをコンテキスト先頭/末尾に配置。関連性の低いノイズを除外 |
| 5 | **x-auto-agent スキーマ非互換** | スキーマファースト設計。JSONL キューのスキーマをバージョン管理 |
| 6 | **日本語トークン消費の未実測** | 初期プロトタイプで実測。コンテキスト上限到達が早い場合はサンプル件数を動的調整 |

### 低優先度リスク

| # | リスク | 対策 |
|---|-------|------|
| 7 | **著作権・ToS グレーゾーン** | 自身のツイートまたは許諾済みのものを優先。個人・少数件・非商用の範囲で使用 |
