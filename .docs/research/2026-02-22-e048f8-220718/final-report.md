## 3. 視点別調査結果

### 3.1 技術的実現性（Technical）

#### レンダリング性能

Alpine.jsは大規模DOMで構造的な性能問題を抱える:

- **3,000 DOM要素**での実測: Alpine有り 11秒 vs なし 3秒（**約3.7倍劣化**）
- 原因: コンポーネント内のいずれかのデータ変更時に**コンポーネント全体を再評価**する設計
- `querySelectorAll`で全要素をイテレートするため、バインディング数に応じて**線形劣化**

**性能順序（推定）**: Svelte > Vue 3 > Petite-Vue > React > Alpine.js

#### Express REST API + SSE統合

| フレームワーク | REST API | SSE (EventSource) | 特記事項 |
|----------------|----------|---------------------|----------|
| Vue 3 | fetch/axios（標準） | vue-sseプラグイン、Composition APIで綺麗に抽象化可能 | 最も成熟 |
| React | fetch/axios（標準） | useEffect + useRefパターン確立 | 情報量最大 |
| Svelte | fetch/axios（標準） | onMount/onDestroyでライフサイクル管理 | ネイティブJS |
| Alpine.js | Fetch API（現行） | x-init/destroyで管理可能だが、コンポーネント間共有が複雑 | $store経由が必要 |

#### 移行コスト（書き換え量）

| 移行先 | コスト | 概要 |
|--------|--------|------|
| **Vue 3** | **最低** | `x-` → `v-` プレフィックス変換でほぼ動作。構文的に最も類似 |
| Petite-Vue | 最低（※未検証） | Alpine.jsと互換性の高いAPI設計。ただしエビデンス不足 |
| Svelte | 中 | .svelteファイル分割 + `{#if}{#each}`構文への書き換え + ビルドツール必須 |
| **React** | **最高** | HTMLテンプレート → JSXへのパラダイム転換。50+メソッドの再実装が必要 |

#### 初期ロードサイズ

| 方式 | Alpine.js | Petite-Vue | Vue 3 | React | Svelte |
|------|-----------|------------|-------|-------|--------|
| Import Maps（ビルドなし） | ~15KB | ~6KB | ~20KB | ~42-45KB（実用困難） | 非推奨 |
| Vite + Tree Shaking | — | — | 最適化可能 | 固定~42KB | コンポーネント数依存 |

### 3.4 代替案・競合（Alternatives）

#### Alpine.js強化路線の限界

Alpine.js ESM完全化 + Components + store() で**部分的にのみ改善可能**。構造的に残る課題:

1. **HTMLテンプレート内ロジック**がユニットテストから隔離不可（alpine-test-utilsは公式サポート外）
2. **SFC（単一ファイルコンポーネント）相当の分離機構**が不在。windowスコープへのグローバル依存
3. **ESMモジュール化時のtree-shaking誤設定**でAlpine依存関係が脱落するリスク
4. **Alpine.store()のネスト状態追跡**が自動でなく、深いリアクティビティで手動処理が増加

#### Web Components（Lit / Stencil）

- フレームワーク非依存性は理論的に明確（Lit: 235ms初期ロード、Stencil: 284ms）
- **しかし**: 管理コンソール大規模導入事例が稀少、SSRが実験段階、Shadow DOMの癖がUI統一性を阻害
- **2026年時点での採用はリスクが高い**

#### ハイブリッドアプローチ（Alpine.js + 別FW混在）

- 技術的に実現可能だが、**実質的なマイクロフロントエンド構築に相当**
- 状態同期（カスタムアダプター層）、イベント橋渡し、ビルドツール複雑化が発生
- フレームワーク混在は**長期的な技術的負債**を生みやすい

#### 管理コンソール特化UIフレームワーク

| ツール | 特徴 | 制約 |
|--------|------|------|
| **shadcn/ui** | Tailwindベース、コピペモデル、~50KB gzip、WAI-ARIA準拠 | **React前提** |
| Tremor | 分析ダッシュボード特化、35+コンポーネント | **React前提** |
| Ant Design Pro | 38k+ Stars、20+ビジネスページ、最大エンタープライズ実績 | **React前提** |
| Refine | ヘッドレス設計、CRUD自動生成、認証・アクセス制御内蔵 | **React前提** |

> **重要**: これらは全てReactエコシステム前提。Alpine.js継続では恩恵を受けられない。Vue向け（Vuetify/Naive UI/Quasar）の比較は未実施。

### 3.5 開発者体験 — DX（Developer Experience）

#### HMR（Hot Module Replacement）

- Viteベースで**理想条件下 10〜20ms**（Vue/React/Svelte共通）
- ただし2000行超コンポーネントではプラグイン処理により**3〜17秒に劣化**する事例報告あり
- フレームワーク間差異よりも**設定・プラグイン依存**が支配的
- Webpack比 500〜1600ms → Viteで 10〜20ms に改善した実例あり

### 3.6 移行継続性（Migration Continuity）

#### 並行運用アーキテクチャ（3パターン）

| パターン | 複雑度 | 特徴 |
|----------|--------|------|
| **パスベースルーティング** | **最低** | `/new/*` → 新UI、既存パス → 旧Alpine.js。Express `express.static()` の複数マウント |
| ストラングラーフィグ + リバースプロキシ | 中 | フィーチャーフラグまたはURLパスで振り分け |
| マイクロフロントエンド (Module Federation) | 高 | 同一ページ内共存。設定の複雑さ増大 |

#### 段階的移行のフェーズ定義

| フェーズ | 対象 | 優先理由 |
|----------|------|----------|
| Phase 1（Week 1-4） | 最も単純な画面（設定、プロファイル） | リスク最小・学習価値最大 |
| Phase 2（Week 5-12） | CRUD中心の画面（一覧、フォーム、ダッシュボード） | 中難易度 |
| Phase 3（Week 13+） | インタラクティブ機能（リアルタイム更新） | 最終段階 |

#### 各フェーズの必須ゲート

- **機能パリティ**: E2Eテスト全件パス、全ユーザーフロー同等動作
- **パフォーマンス**: LCP・FID・CLSが旧UI比で劣化なし
- **運用**: フィーチャーフラグによる即時切り戻し確認済み、モニタリング正常値24-72時間

#### ロールバック戦略

| 手法 | 所要時間 | 特徴 |
|------|----------|------|
| **フィーチャーフラグ切り戻し** | **数分** | サーバー設定変更のみ。**最推奨** |
| ブルーグリーン/カナリアリバート | 30分〜2時間 | トラフィック比率変更。コードデプロイ不要 |
| Gitリバート + 再デプロイ | 30分〜4時間 | 画面独立なら対象画面のみ |
| フレームワーク全体撤退 | 数日〜数週間 | 移行進捗に比例 |

**フレームワーク別ロールバックコスト**: Vue 3（最低）> React（中）> Svelte（高：エコシステムギャップによる埋没コスト）

## 5. 推奨事項

### 5.1 主要推奨（Primary）

> **d-20260221-135800のPhase Aを実行**（Alpine.js ESモジュール完全化、20-40h）
> Phase B（API移行）完了後にPhase C（UI技術判断）を実施。
> **フルリデザインは現時点で着手しない。**

**エビデンス強度**: 6視点中5視点以上が直接的または間接的に支持

**残余リスク**:
- Phase AはDX課題（TypeScript統合ゼロ・DevTools品質）を解決しない
- UI/UXデザイン調査が未実施
- Phase Aの投資がPhase CでFW移行決定時に部分的に無駄になる可能性
- Phase Cの開始時期がAPI移行進捗に依存

### 5.2 フォールバック推奨（Fallback）

> Phase A完了後 **かつ** Phase B（API移行）完了後に、**Vue 3へのStrangler Pattern段階的移行**を実施

**トリガー条件**: Phase A完了後に以下の5指標を測定し、**3つ以上が未達**の場合に発動

| # | 指標 | 閾値 |
|---|------|------|
| 1 | コード探索時間 | 任意のメソッド所在特定が **60秒以内** に完了しない |
| 2 | テストカバレッジ | 分割後モジュールの単体テストカバレッジが **40%** に到達不可能 |
| 3 | レンダリング性能 | メイン画面のLighthouse Performance Scoreが **60未満** |
| 4 | 状態管理複雑性 | Alpine.store()導入後もグローバル状態プロパティが **25以下** に削減不可能 |
| 5 | 新機能追加速度 | 新セクション追加が **16h以上** かかる |

**Vue 3を選ぶ理由**:
- Alpine.js経験者に最低移行コスト（構文類似性、学習1-3日）
- Volarによるテンプレート型チェック最高水準
- Vue DevTools v7が機能最充実
- プログレッシブ設計でロールバック最容易

### 5.3 ABORT判定（フルリデザイン中止）

**API移行完了前のフルリデザイン着手を中止する。**

根拠:
