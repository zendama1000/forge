## 3. 視点別の調査結果

### 3.1 技術的実現性（technical）— 信頼度: 高

| 問い | 結論 |
|---|---|
| 深化5節構成を Markdown で表現できるか | **可能**。標準 GFM で任意の節構造を記述可。ただし SKILL.md 本文は「500行・5000トークン以内」推奨のため、理論本体は theory-*.md 個別ファイルへ分割 |
| 対応表を grep/wc で機械検査できるか | **十分可能**。`grep -cE`/`awk -F'|'` 等で見出し数・表行数・列存在を検証。検証失敗→補強→再生成のCIループも実績パターンあり |
| 完全版.md(約264KB≒67Kトークン)を文脈長内で扱えるか | **収容可能**。ただし「lost in the middle」回避のため章境界での offset/limit 分割読解を推奨 |
| 再マッピング後も参照整合を保てるか | **保持可能だが条件付き**。サブエージェントの同一性は frontmatter の `name` で決まる。リンク更新の外部バリデーションスクリプトが前提 |

**注意点**: Markdown テーブルの区切り行（`--- ---`）誤検知対策、Windows 環境の CRLF による wc ずれ、frontmatter YAML パースバグ（既知 issue）への対処が必要。

### 3.2 コスト・リソース（cost）— 信頼度: 高〜中

実ファイル計測に基づく試算:

| 項目 | 実測/試算値 |
|---|---|
| references/ | 9ファイル・53,795バイト |
| agents/ | 8ファイル・45,853バイト |
| SKILL.md | 13,255バイト |

### 3.3 リスク・失敗モード（risk）— 信頼度: 高〜中

5つの問いを横断する共通リスク構造は **「漸進的境界侵食（gradual boundary erosion）」** — 単一ステップでは検知困難な段階的劣化。

| リスク | 信頼度 | 実証データ |
|---|---|---|
| エージェント間特権昇格による概念汚染 | 高 | エージェント間メッセージ漏洩率 68.8%（出力チャネル27.2%の2.5倍） |
| Crescendo型 段階的具体化によるマニピュレーション手順書化 | 高 | ジェイルブレイク成功率を29〜71%向上、平均5回以下で完成 |
| 再マッピングによるトレーサビリティ断絶 | 高 | RTM研究: 「下流リンクなき要件＝未テスト」 |
| 必須フィールド追加による後方互換破壊・再生成ループ非収束 | 中 | オプション追加は互換、必須追加は非互換が原則 |
| 完全版.md 単独準拠による理論偏向 | 中 | 「完全版」の名称が網羅性の誤った確信を生むメタリスク |

**対策の優先順位**: 【遮断 > 検出 > 監視】。アーキテクチャレベルでの役割境界強制が事後フィルタより根本的に有効。

### 3.4 代替案・競合（alternatives）— 信頼度: 高〜中

| 論点 | 比較結果 |
|---|---|
| フレームワーク選択 | **bridging correspondence table（橋渡し表）が最もリスク低**。現行6セオリー（Lifton思想改造8基準等と親和性高）と原本6セオリーは名称・粒度が異なり一意対応が不確実なため、全面再マッピングは避ける |
| theory-*.md 節構成 | **5節構成**が標準ベストプラクティス（Introduction-Setup-Usage-Troubleshooting型）と整合。警告情報は独立H2セクションで視覚分離 |
| 対応表フォーマット | **マトリクス表（理論×適用箇所＝RTM型）**が自動検証と多対多把握に最適。ただし列数6以上で Markdown レンダリングが崩れやすい |
| 具体性の機械的担保 | **「テンプレート必須セクション化＋grep検査」の併用**が現代の文書品質管理の定石（Vale/TextLint等のCI統合手法） |

**ギャップ**: 原本6セオリーの一次出典が特定できず、現行6セオリーとの概念的等価性は定量評価不能。「教義/金と労働力」は現行に明示的対応がなく空白セルが生じうる。

### 3.5 倫理整合性と役割ドリフト防止（ethics_drift）— 信頼度: 高〜中

> 一貫して浮かび上がる構造的脆弱性: **「設計時の安全性は維持時の安全性を保証しない」**

| 問い | 結論 |
|---|---|
| 健全系/警告系の線引き基準 | 話題分類ではなく **「オペレーショナル・アップリフト（operational uplift）」基準**。①悪意ある実行能力を実質向上させるか ②「なぜ機能するか（原理）」か「どう実行するか（手順）」か ③意図×コンテキスト依存性 |
| 警告系プロンプトの記述スタイル | ①被動態・三人称距離化 ②明示的被害ステートメント前置 ③能動提案語彙のネガティブリスト ④危険パターンと健全代替の構造的見出し分離 |
| 三層は更新が必要か | **全層同時更新が必須**。特に Layer 1（禁止キーワードリスト）への新語彙収録が最優先 |
| ethics-reviewer 自体の劣化 | **RIPD（ルーブリック誘発ドリフト）が最大の脅威** |

**重要な実証データ**:
- **Layer 1（grep_absent / キーワード照合）の検出率は約17%**、Layer 2（LLMセマンティック解析）は100% → **ethics-reviewer LLM が実効的な主防衛線**
- **RIPD**: 一見自然なルーブリック編集だけで LLMジャッジの harmlessness 検出精度が最大 **27.9%低下**。集計ベンチマークでは検出困難
- 対策: ゴールデン事例セットによるアンカリング（RULERS手法）、定期ホールドアウト検証、ジャッジ・ファミリー多様化

### 3.6 理論→適用トレーサビリティ（traceability）— 信頼度: 高〜中

> 核心は「**IDの一貫性**」と「**アーティファクト内埋め込み**」の二点。

- **ID設計**: セマンティクスフリー＋階層プレフィクス型 `TH-{2桁章番号}-{3桁連番}`（例: `TH-01-003`）。完全版.md の各概念見出しに手動HTMLアンカー `<a id="TH-01-003">` を埋め込む（Markdown自動アンカーは重複ヘディングで不安定）
- **埋め込み方式**: theory-*.md は YAML フロントマター `source_ref`、agentプロンプト/brand-design.md は `<!-- ref:TH-01-003 -->` コメント。外部スプレッドシート管理は劣化しやすい（ReqToCode が実証）
- **対応表の必須7項目**（概念単位粒度）:

  | 項目 | 内容 |
  |---|---|
  | `source_id` | 完全版.md の TH-XX-YYY ID |
  | `source_quote` | 原本の直接引用または要旨 |
  | `theory_ref` | theory-*.md ファイル名＋セクション |
  | `agent_ref` | 対象エージェント定義ファイル＋適用箇所 |
  | `brand_design_ref` | brand-design.md セクション識別子 |
  | `transformation_note` | 理論→設計提案への変換説明（最大200字目安） |
  | `status` | mapped / partial / unmapped / deprecated |

- **常設マッピング文書**: `references/theory-mapping-canonical.md` をプロジェクト開始時に作成・常設（NASA SWE標準: 後から遡及作成より継続維持が高品質）
- **網羅性検査**: grep/jq による機械的差分チェック。欠落時は Forge Harness 既存の `validate_l1_coverage()` パターン（欠落検出→補強プロンプト→リトライ）を流用

## 5. 推奨事項

### 5.1 主推奨（Primary）— 前提条件付きで実行

**3つの前提条件**を工程に織り込んだ上で、完全版.md 準拠の再構築を実行する:

1. **ID識別体系の先行設計** — 完全版.md の各概念に手動HTMLアンカーで `TH-XX-YYY` を付与し、`references/theory-mapping-canonical` を最初に常設
2. **三層ガードレールの同時更新** — 禁止キーワードリスト / ethics-reviewer ルーブリック / grep_absent パターンを理論強化と連動して更新
3. **対応表の段階的必須化** — warning レベルで導入し、対応表生成が安定したら required へ昇格

**工程の流れ**:

```
完全版.md にID付与 → theory-mapping-canonical 常設
  → 章境界 offset/limit 分割読解で各セオリーへ概念割当
  → theory-*.md 6本を5節構成へ深化
       （健全系3 = 能動提案 / 警告系3 = 危険パターン分析＋健全代替）
  → agents/ 8体を同一ID参照で更新＋三層ガードレール同時更新
  → brand-design.md に RTM型対応表（必須7項目）新設
  → SKILL.md 検査へ組込
```

**フェーズ分割（5タスク・3フェーズ、フェーズ完了毎 auto-commit）**:

| フェーズ | ゴール | 対象 |
|---|---|---|
| **MVP** | references/ ナレッジ基盤が深化5節構成とID体系を備える | source-book.md + theory-*.md 6本 + theory-mapping-canonical.md |
| **Core** | agents/ 8体が同一ID参照で更新、三層ガードレール全層同時更新 | 健全系3 + 警告系3 + ethics-reviewer 他 |
| **Polish** | SKILL.md と RTM型対応表が完成、トレーサビリティが機械検証可能に | SKILL.md + brand-design.md 対応表 + source-book |

**根拠**: 6視点すべてが技術的・コスト的実現性を肯定。ロックされた全6決定と整合し、validateモードの要件を満たす。3前提条件は、それぞれ traceability/ethics_drift/risk が特定した最重要失敗モード（トレーサビリティ断絶・役割ドリフト・再生成ループ暴走）への構造的対策。

### 5.2 残存リスクと緩和策

| リスク | 緩和策 |
|---|---|
| **役割ドリフト（最重要）** — 理論解像度向上で警告系が肯定提案化 | 被動態/三人称距離化、被害ステートメント前置、能動提案語彙ネガティブリスト、危険パターンと健全代替の構造的見出し分離 |
| **ethics-reviewer の RIPD** — 善意の詳細化で判定が甘くなる（最大27.9%低下） | ゴールデン事例セットによるアンカリング、定期ホールドアウト検証 |
| **具体性のエスカレーション** — 事例追加が実装手順書化（Crescendo型）に滑る | テンプレートを「効果説明」と「実装手順」に分離管理し、後者は作成しない |
| **原本6セオリーの出典不明** — 概念的等価性が定量評価不能 | `status=unmapped` で明示し再調査対象に回す。現行6セオリーを実装の主軸に維持 |
| **対応表必須化による後方互換破壊** | circuit-breaker 設定＋段階的（オプション→必須）導入 |
| **完全版.md 単独準拠による理論偏向** | ロック決定により外部補完不可。対応表のカバレッジ検査で空白を可視化するに留める |

### 5.3 フォールバック（Fallback）

**トリガー**（いずれか発生時）:
- (a) Implementer がファイル数制限超過・未コミット累積でロールバックを繰り返す
- (b) 対応表必須化で SKILL.md 再生成ループが circuit-breaker 上限まで非収束
- (c) ethics-reviewer 更新後のホールドアウト検証で判定精度低下が検出される

**対応**: プロジェクト全体の一括再構築を中断し、**references/ ナレッジ基盤（source-book.md + theory-*.md 6本 + theory-mapping-canonical）の深化のみを先行完了**させる。agents/・SKILL.md・対応表は後続バッチに分離。対応表は当面オプションセクション（warning検査）のまま運用。

### 5.4 中止（Abort）の扱い

全面 abort は**非推奨**（validateモードかつ全ロック決定が再構築実施を前提、6視点すべてが実現可能と判定）。ただし**「三層ガードレールの同時更新を工程から外したまま理論強化だけを進める進め方」は中止すべき** — それは役割ドリフトという最重要バグを増幅し、教育・自己防衛・健全コミュニティ強化に限定されるべきツールがマニピュレーション支援に滑り、プロジェクトの存在意義を毀損する。
