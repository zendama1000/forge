## 3. 検証済み・問題なしの設計要素

### 3.1 Phase別モデルルーティング

Agent toolの`model`パラメータ（enum: sonnet/opus/haiku）で正式サポート。サブエージェント定義ファイルのYAMLフロントマターでも`model: haiku|sonnet|opus|inherit`が指定可能。

| Phase | モデル | 根拠 |
|-------|--------|------|
| A（メタフレーム抽出） | Opus | GPQA Diamond: Opus 91.3% vs Sonnet 74.1%（+17pt）。複雑な理論体系理解に必要 |
| B（アウトライン+セクション） | Sonnet | 構造化生成タスクではOpusとの差が1.2ptと最小 |
| C（統合+商品注入） | Sonnet | 同上 |
| D 形式チェック | Haiku | Sonnet比70%コスト削減。分類・パターンマッチに近いタスクで同等精度 |
| D 意味評価（初稿） | Sonnet | コスト効率の初段評価 |
| D 意味評価（最終） | Opus | 高精度な最終判定（Sonnet閾値+5pt以上ならスキップ可） |

**注意**: Issue #29768（Exploreサブエージェントがhaiku指定を無視）の修正状況未確認。実地検証が必要。

### 3.2 ハイブリッド状態管理（複数JSONファイル）

| 管理方式 | 原子性 | 可読性 | 復旧容易性 |
|----------|--------|--------|------------|
| **複数JSON（採用）** | 中（tmp+rename） | **最高** | **最高**（ファイル単位復旧） |
| 単一大JSON | 低（全体破損リスク） | 中 | 低 |
| SQLite | **最高**（WAL/journal） | 低（外部ツール必要） | 中 |

LLMパイプラインの並行性が低い環境では、SQLiteの原子性優位は縮小する。

### 3.3 SKILL.mdによるリライトループ外部制御

- Skills 2.0（2026年3月更新）でループ+閾値ゲートパターンが公式サポート
- 研究エビデンスが外部制御を支持: 自己完結リファインはreward hackingリスク+自己エラー検出の弱さ
- 実装パターン: Phase-Cエージェント→Phase-Dエージェント（スコアJSON返却）→SKILL.mdがRead→条件分岐→再呼出

### 3.4 その他の検証済み要素

- **Phase-A理論ファイル全文注入+以降metaframeのみ**: 効率的。6本程度なら問題なし
- **pre-flightバリデーション**: SHA-256チェックサム+FIMパターンで防御可能
- **JSONスキーマ検証層**: PostToolUseフック（フィードバック注入）+SKILL.md（カウンタ制御）で実装可能
- **Phase-D Haiku形式チェック**: 70%コスト削減、同等精度で最もリスクの低い最適化

## 4. 修正推奨3点の詳細

### 4.A セクション生成: 並列→順次（overlap_context付き）

**問題**: 3視点が一致して並列化の品質リスクを指摘

| 軸 | 並列生成+seam adjustment | 順次生成+overlap_context |
|----|--------------------------|------------------------|
| **品質** | 感情的フロー23.6%劣化（SCORE framework）、Desire→Actionの累積感情喪失 | overlap_contextが文脈連続性を直接緩和 |
| **速度** | 1.8x高速 | 基準（1.5-2x遅い） |
| **コスト** | 1.8xコスト増 + Tier 1 ITPM=30,000で即座超過 | 標準コスト |

**推奨実装**: Agent toolを1セクションずつ順次呼出。前セクション末尾500-1000トークンをoverlap_contextとして次セクションに注入。Phase-Cのseam adjustmentステップは削除。

**フォールバック**: 速度が問題になった場合、WritingPathハイブリッド方式（スケルトン先行→2-3セクション単位バッチ並列→post-process統合）を検討。

### 4.B 品質評価: Sonnet 2回平均→Sonnet+Opus非対称クロス評価

**問題**: 2つの環境変化が既決設計を陳腐化

| 要因 | 詳細 |
|------|------|
| Sonnet自己検出バイアス | 評価シナリオを58%の確率で検出し行動を異常良化。2回平均はバイアス増幅 |
| Opus価格改定 | 旧$15/$75 → 現$5/$25（67%値下げ）。差は1.67倍に縮小 |

**推奨実装**:
1. Phase-D形式チェック → Haiku（維持）
2. Phase-D意味評価 → Sonnet初稿評価
3. Sonnet評価が閾値+5pt以上 → Opus評価スキップ（コスト最適化）
4. 閾値付近 → Opus最終判定

**追加コスト**: 単一Sonnet比約2.3倍（Sonnet 2回の2倍との差はわずか0.3倍）

### 4.C リライトカウンタ: SKILL.md内→state/phase-state.jsonに永続化

**問題**: SKILL.md内テキストカウンタは揮発性。ralph-loop 646回ループバグ（Issue #18086）が実証済み先例。

**推奨実装**:
- `phase-state.json`に`rewrite_count`, `last_score`, `score_history`フィールドを追加
- SKILL.mdは毎Phase-D開始時にReadで読み取り
- **複合停止条件**: max 2回 **AND** 最小デルタ収束判定（前回比+2pt未満で収束扱い）
- 閾値未達でmax到達時は`human_review: true`で正常終了

## 7. 視点間の矛盾と解決

### 7.1 セクション並列化の評価分裂

| 視点 | 評価 |
|------|------|
| Technical | 技術的に実現可能（ファイル分離で競合回避） |
| Content Quality | 感情的フロー23.6%劣化、seam adjustmentでは補正困難 |
| Cost | Tier 1で即座に上限到達 |
| Alternatives | 品質軸で順次生成が優位、速度軸のみ並列が1.8x高速 |

**解決**: セールスレター生成の本質的価値は出力品質。速度は副次的。**順次生成を採用**（Agent toolの順次呼出で実現、ロック決定事項と矛盾なし）。

### 7.2 品質評価方式の環境変化

既決の「Sonnet 2回平均化」は旧来の5xコスト差を前提とした設計。Opus 4.6の価格改定（1.67倍差）+Sonnetのバイアス報告により陳腐化。**Sonnet+Opus非対称クロス評価**に変更。

### 7.3 商品注入のPhase統合 vs 独立化

Content Quality視点は独立Phase化を推奨、Alternatives視点は厳密なスキーマ定義で統合可能と判断。**Phase-C内統合を維持**し、品質問題顕在化時のみ独立化を検討する段階的アプローチを採用。

## 11. Abort条件

以下いずれかに該当する場合、移行を中止し現行Express+TypeScriptパイプラインを維持:

1. Agent tool `model`指定が致命的バグで全Phaseで同一モデルしか使えない

3. スキルベース統合テストの確立が不可能で、テストカバレッジが著しく低下する

**中止時の機会損失**: Claude Code内シームレス利用・Phase制御による責務分離・モデルルーティング柔軟性が全て失われ、llm-service.tsのClaude CLI子プロセス呼出という技術的負債が残存。
