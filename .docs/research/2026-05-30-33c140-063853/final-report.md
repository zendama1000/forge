## 1. テーマと問い

Claude Code の **Dynamic Workflow** 機能（2026-05-28 リリースのリサーチプレビュー）を、**無人デタッチ自律性を損なわない範囲で**、現行 Forge Harness（bash + `claude -p` の自前マルチエージェント・オーケストレーション）に**補完的に**活用できるか。適用箇所・採用是非の判断材料・移行/PoC 計画を設計ドキュメントとして示す。

### コアとなる問い

1. どの内側コンポーネント／有人フェーズなら、デタッチ無人性を損なわず Workflow を投入でき、費用対効果が最大か
2. `claude -p` がサブエージェント／Task 機構をロードしない制約を、無人デタッチ・プロセス跨ぎ再開を保ったまま回避できるか
3. Workflow 補完の費用対効果は、既存堅牢化資産の再実装量に見合うか
4. Routines（クラウド定期実行）や Agent SDK は無人デタッチ・長時間・プロセス跨ぎ再開とどこまで両立するか
5. Workflow の制約（JS のみ／I/O は agent 経由／`Date.now`・`Math.random` 不可／人手入力不可／セッション内のみ resume／最大16同時・1000累計）がハーネスの決定論的制御・状態永続・再開・観測性にどう影響するか

### ロック済み（調査対象外）の前提

- **コア自律ループ**（forge-flow / research-loop / ralph-loop）の Workflow による**置換は対象外**
- **デタッチ無人・プロセス跨ぎ再開を損なう用途は対象外**
- 成果物は**設計／判断ドキュメントのみ**（本体実装は行わない）

## 2. 結論（エグゼクティブサマリー）

> **6 視点すべてが、「Workflow をコアの置換ではなく、Phase 1 リサーチ fan-out（SC→R×6→Syn→DA）への限定的補完候補として設計ドキュメント化するに留める」方向に収束した。**

- **技術的には成立する**: 6 並列は 16 同時上限内、ファイル I/O は `agent()` 経由で橋渡し可能、`Date.now`／人手入力不可は `args` 注入と Phase 0/4 の外部境界化で回避可能。
- **しかし費用対効果は限定的**: 堅牢化資産（`.pending` 昇格・circuit-breaker・lessons 注入）はエージェント／シェル層に残るため再実装コストは節約されず、〜10 並列規模ではトークン利得も軽微。
- **3 つの根本制約が high 確信度で共通指摘された**（後述）。
- したがって **「実装には進まず、最小 PoC で検証可能な設計ドキュメントを揃える」**のが推奨。

## 4. 共通して指摘された 3 つの根本制約

### ① セッション内 resume 限定 × デタッチ無人再開と非両立
- Workflow の `resumeFromRunId` は**同一セッション内のみ**。CLI を終了すると次セッションで「最初から」再実行。
- **daemon-spawned セッションでは deferred tools（Task）が未ロード**という既知バグ（GitHub Issue #111, #19077, #25200 等、MEMORY.md 2026-04-12 実地検証とも整合）。
- → `nohup --daemonize` 相当のプロセス跨ぎ無人再開と**両立しない**（＝ロック決定がまさにこれを除外している）。

### ② state 二重管理による不整合・観測性低下
- 既存の `.forge/state/`（task-stack.json, progress.json）と Workflow の journal（`agent-<id>.jsonl`）は**別系統で同期機構がない**。
- stale session 残留で「Session ID already in use」、設定値のグローバル漏れなどの事例あり。

### ③ 2026-06-15 以降の SDK／Routines クレジット制移行
- Pro $20／Max 5x $100／Max 20x $200・**ロールオーバーなし**、フル API レート課金。
- Opus ヘビーの自律ループは $200 を 36〜72 時間で消費しうる → 枯渇で即中断。
- Managed Agents は Claude 専用で**ベンダーロック**（他モデル移行機構なし）。

## 5. 矛盾点と解消

| 対立 | 内容 | 解消 |
|---|---|---|
| **代替案 ⇄ コスト** | 代替案は Workflow のネイティブ並列・adversarial verify・resume を構造的優位として PoC を推す。コストは堅牢化資産が残り再実装が節約されず、〜10 並列では利得軽微と懐疑的 | **適用規模で和解**。Workflow の優位は 50+ エージェント規模で効く。Forge の 6 並列では薄い。よって「全面採用は薄い（コストが正）」が「最小 PoC で adversarial verify の品質改善のみ限定検証する価値はある（代替案が正）」。PoC は fan-out 1 箇所に絞り、堅牢化資産は bash 層に温存 |
| **技術 ⇄ リスク** | 技術は「6 並列への限定適用＋daemonize 維持のハイブリッド」を建設的に評価。リスクは resume 限定・state 二重管理・クレジット枯渇を強調 | **前提の差で矛盾せず**。両者とも「Phase 0/4 を外部境界化し daemonize は bash+tmux 維持」という同じ隔離設計が前提。リスクの懸念はコアに織り込んだ場合に顕在化する。PoC を別セッション・別プロセスで隔離し、産出物のみ `append` 経由でコアに渡す設計要件として吸収 |

## 6. 推奨アクション（3 段階）

### ✅ 主推奨（Primary）

Workflow は **「Phase 1 リサーチ fan-out（SC→R×6→Syn→DA）の補完候補」として設計ドキュメント化に留め、コア自律ループには一切織り込まない**。実装・本体コード変更は行わない。成果物は以下 4 点：

1. **責務分離アーキテクチャ** — Workflow=セッション内並列処理／コアループ=ファイル状態機械、Phase 0/4 は Workflow 外部境界
2. **Strangler Fig 型 PoC 設計** — `research.json` フィーチャーフラグによる投入／無痕跡撤退
3. **観測連続性** — 産出物は既存 `append_decision()` 経由で JSONL append-only
4. **段階的 go/no-go ゲート**（下表）

**根拠**: ロック決定（コア置換除外・ドキュメントのみ）に完全整合。全 6 視点が research fan-out を最小リスク・最高費用対効果の唯一の補完候補として収束支持。技術的成立性が high 確信度で確認済み。

**残るリスク**:
- リサーチプレビュー段階で API 変更 → ドキュメント陳腐化
- 堅牢化資産が再実装節約に効かず、PoC 実装に進んでも費用対効果が想定割れの可能性
- daemon-spawned Task 未ロードバグ・クレジット制移行など外部要因で前提崩壊

### 🔄 フォールバック（Fallback）

Workflow 補完を見送り、**adversarial verify（独立反証エージェントの多数決）を現行 bash + `claude -p` 並列上に自前実装**する設計ドキュメントを代替成果物とする。DA フェーズの反証ロジックを既存 research-loop に組み込み、品質改善のみ取り込む。

- **トリガー**: PoC 設計段階で (a) Workflow journal と `.forge/state/` の競合が解決不能、(b) クレジット制でコスト上限が運用規模に不足、(c) Workflow API が GA 前に破壊的変更、のいずれか確認時。

### ⛔ 中止（Abort）

現行 bash + `claude -p` ハイブリッドは制御性・コスト予測性・デタッチ無人性・堅牢化資産すべてで依然有力。〜10 並列規模では Workflow の構造的利得が顕在化しない。**現状維持し、再評価を Workflow GA 後まで延期**するのも合理的。

- **機会費用**: 組込 adversarial verify・コンテキスト外オーケストレーション・セッション内 resume の早期取り込みを逃す。ただし fan-out が 50+ エージェントに拡大した時点で再評価すれば取り戻せ、機会費用は限定的。

## 8. 実装基準（フェーズ計画の概要）

`implementation-criteria.json` には、成果物検証のため L1（静的）／L2（統合）／L3（LLM/E2E）基準と 3 フェーズが定義されている。

| フェーズ | ゴール | 主成果物 |
|---|---|---|
| **mvp** | 責務分離設計＋go/no-go ゲート定義が成立、ロック決定整合を確認 | `responsibility-separation.md`, `gates.json` |
| **core** | フラグ・PoC スクリプト・撤退設計が揃い ON/OFF 分岐と append-only 観測が動作 | `research.json`（既定 OFF フラグ）, `research-fanout.workflow.js`（meta 純リテラル・6 並列）, `rollback.md` |
| **polish** | PoC が SC→R→Syn→DA 連鎖を模擬、Synthesizer 出力がスキーマ適合、撤退後に状態が無痕跡回帰 | smoke 実行検証・スキーマ機械適合・最終整合確認 |

主要な検証ゲート例:
- **L1**: 設計ドキュメント必須セクションの grep＋語数閾値、`gates.json` の数値型チェック、PoC スクリプトの `node --check`＋禁止パターン（`Date.now`/`Math.random`/`${`）検出、フラグ既定 OFF の型検証
- **L3-001**: ロック決定遵守・責務分離・撤退実行可能性・観測連続性を LLM が 4 観点でスコアリング（閾値 0.8）
