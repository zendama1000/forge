## 3. 各視点の調査結果

### 3.1 技術的実現性（technical）

#### T1: LLM JSON出力安定性

- **Ollama**: `format`パラメータでJSON schema指定可能。複雑スキーマでは不完全JSON・コメント混入のケースあり
- **vLLM**: XGrammar/Outlines/lm-format-enforcerの3バックエンド。XGrammarが最も安定
- **LM Studio**: OpenAI互換JSON schema強制サポート。詳細な信頼性データ少ない
- **推奨フォールバック戦略**: temperature=0 + 明示的JSONプロンプト指示 + max 3回リトライで実用的安定性
- **信頼度**: 中（モデル固有の失敗率データは非公開）

#### T2: SSEストリーミング × Vercel制約

| 制約項目 | 詳細 |
|---------|------|
| タイムアウト | Edge Runtime: 300秒（2025年3月以降）。**最初のレスポンスを25秒以内に送信開始する必要あり** |
| バッファリング | `export const dynamic = 'force-dynamic'` + `ReadableStream`パターンが必須 |
| 接続切断検知 | `request.signal.aborted`でAbortControllerパターンを使用 |

- **信頼度**: 高

#### T3: アニメーション手法比較

| 手法 | バンドルサイズ | モバイルパフォーマンス | DX |
|------|-------------|---------------------|-----|
| CSS 3D Transform | **0KB** | GPU加速で最高 | 記述量多い |
| Framer Motion (LazyMotion) | 4.6KB〜32KB | 複数同時で45fpsに低下報告 | 直感的API |
| Motion One (WAAPI) | 3.8〜4KB | ネイティブ最大120fps | 中間 |

- **推奨**: フリップ単体 → CSS 3D。複雑シーケンス → Framer Motion LazyMotion構成
- **信頼度**: 高

#### T4: レート制限 × Edge Runtime

- Edge Runtimeはインメモリ状態保持不可（サーバーレスのステートレス性）
- **実質唯一の選択肢**: Upstash Redis（HTTP REST API）+ `@upstash/ratelimit`
- 固定ウィンドウ/スライディングウィンドウ/トークンバケットから選択可能
- **信頼度**: 高

#### T5: カードDB型安全設計

```typescript
// Zod discriminatedUnionパターン
const MajorArcanaSchema = z.object({
  arcana: z.literal('major'),
  number: z.number().min(0).max(21),
  name: z.string(),
  upright: z.string(),
  reversed: z.string()
});

const MinorArcanaSchema = z.object({
  arcana: z.literal('minor'),
  suit: z.enum(['wands','cups','swords','pentacles']),
  rank: z.union([z.number().min(1).max(10),
                 z.enum(['page','knight','queen','king'])]),
  upright: z.string(),
  reversed: z.string()
});

const CardSchema = z.discriminatedUnion('arcana',
  [MajorArcanaSchema, MinorArcanaSchema]);
```

- **注意**: `z.discriminatedUnion`は`z.switch`へのdeprecation議論中（まだ非推奨ではない）
- **信頼度**: 高

### 3.2 コスト・リソース（cost）

#### C1: アニメーションライブラリのコスト

| 項目 | Framer Motion (full) | Framer Motion (LazyMotion) | CSS only | TailwindCSS Motion |
|------|---------------------|---------------------------|----------|-------------------|
| バンドル | ~32KB gzip | 4.6KB初期 | 0KB | ~5KB |
| 開発工数 | 低（直感的API） | 低 | 高（記述量多い） | 中 |

**結論**: LazyMotion利用でバンドルコスト許容可能。CSS-onlyへの移行で節約できるバンドルサイズ（~27KB）より実装工数削減の方が経済的価値が高い。

### 3.4 代替案・競合（alternatives）

#### A1: JSON修復ライブラリ比較

| ライブラリ | Stars | 最終更新 | 用途 | 推奨度 |
|-----------|-------|---------|------|--------|
| **jsonrepair** (v3.13+) | 2.1k | 2ヶ月前 | 汎用JSON修復 | **最推奨** |
| partial-json (v0.1.7) | - | 2年前 | SSEストリーム途中JSON | SSE用に併用 |
| dirty-json (v0.9.2) | - | 5年以上前 | - | **非推奨（廃棄候補）** |

#### A2: アニメーションライブラリ比較

| ライブラリ | バンドル | 60fps安定 | ライセンス | タロット適合度 |
|-----------|---------|-----------|-----------|-------------|
| CSS @keyframes | 0KB | 最高（GPU） | - | フリップ単体に最適 |
| **GSAP** v3.x | ~23KB gzip | 最高（複数同時OK） | **完全無料**（2024年〜） | **シーケンス演出に最適** |
| Framer Motion | ~32KB gzip | 複数同時で低下 | MIT | API直感的だが互換性リスク |

**結論**: GSAP完全無料化（2024年4月Webflow買収）により、ライセンス障壁が消滅。タロットカードシーケンスにはGSAP timeline APIが最有力。

#### A3: レート制限比較

| 手法 | Vercel無料枠 | Edge Runtime対応 | 推奨度 |
|------|-------------|-----------------|--------|
| インメモリMap | サーバーレスで不可 | 状態保持不可 | 本番非推奨 |
| Vercel WAF | 基本機能のみ含む | 細粒度制御限定 | 補助的 |
| **Upstash Redis + @upstash/ratelimit** | **10,000 req/day** | **HTTP REST API対応** | **最推奨** |

#### A4: ローディング演出比較

| 手法 | 神秘感 | バンドル | 実装コスト |
|------|--------|---------|-----------|
| Skeleton UI | 低（ユーティリティ的） | 0KB | 低 |
| **カスタムシャッフルアニメ** (GSAP) | **最高** | 既存GSAP利用 | 中 |
| Lottie | 中 | ~60.5KB gzip | 低 |

#### A5: LLM JSON安定化手法

| 手法 | 信頼性 | OpenAI互換API対応 |
|------|--------|------------------|
| Structured Output (strict: true) | ~100% | **strictパラメータ無視される** |
| JSON mode | 有効JSON保証（スキーマ非保証） | 対応 |
| プロンプトエンジニアリング + jsonrepair | 最低（フォールバック） | 対応 |

**重要な制約**: ロック決定「OpenAI互換API抽象化層」経由ではstructured outputの信頼性が大幅に低下する。4層防御で補償が必要。

## 4. 統合分析・矛盾と解決

### 視点間の矛盾と解決策

| # | 矛盾する視点 | 内容 | 解決策 |
|---|-------------|------|--------|
| 1 | technical ↔ alternatives | 複雑アニメーション推奨の不一致（Framer Motion vs GSAP） | **GSAP採用**。60fps安定 + App Router互換性問題なし + 完全無料化 |

| 3 | alternatives ↔ ロック決定 | Native Structured Outputs（最高信頼）vs OpenAI互換API（ロック） | ロック尊重 + **4層防御**で補償。将来のnative API移行時に修復層を段階除去 |
| 4 | cost ↔ ロック決定 | Vercel Hobby非商用制約 vs スクール導線CTA（商用意図） | MVP開発はHobby → 本番前にPro移行（$20/月） |
| 5 | risk ↔ ux_mystique | ダークUI + ゴールドのa11yリスク vs 神秘的体験設計 | WCAG AA準拠ゴールド（#FFD700系）を採用。装飾はa11y適用外 |
| 6 | content_integrity ↔ ux_mystique | バーナム効果回避 vs 体験の心地よさ | 具体的行動喚起で品質と満足度を両立。バーナム効果はLLM差別化にならない |

### 7つの統合テーマ

1. **LLM JSON安定性** — 4視点が収束: jsonrepair + partial-json + Zodバリデーション + リトライの4層防御
2. **アニメーション戦略** — CSS 3D（フリップ単体）+ GSAP timeline（シーケンス演出）
3. **Vercel制約** — Edge Runtime SSE 300秒。25秒以内のfirst response必須
4. **レート制限** — Upstash Redis + @upstash/ratelimit（実質一択）
5. **カードDB品質** — LLM生成コスト極小（$0.39〜$0.77）、人手レビューがボトルネック（5.6〜9.3h）
6. **UX体験設計** — タイプライター効果 + カードフリップ後のワンテンポ間 + アニメーションON/OFF
7. **前回決定との関係** — 別プロジェクトとして独立。汎用パターンのみ継承

## 5. 推奨アクション

### Q1: LLM JSON安定化 → 4層防御パイプライン

```
OpenAI互換API (JSON mode有効化)
  → jsonrepair (v3.13+) による修復
    → partial-json (v0.1.7) によるストリーム途中JSON対応
      → Zod discriminatedUnion による型検証
        → max 3回リトライ (フォールバック)
```

client.tsに`repair → validate → retry`パイプラインとして実装。temperature=0デフォルト。

### Q2: アニメーション → CSS 3D + GSAP

| 用途 | 手法 | 理由 |
|------|------|------|
| カード単体フリップ | CSS 3D Transform (rotateY + perspective) | GPU最適、0KB |
| スプレッド展開シーケンス | GSAP timeline (v3.x, ~23KB) | 60fps安定、App Router互換性◎ |
| シャッフル演出 | GSAP timeline | カスタムアニメで神秘感最大 |

Framer Motionは**不採用**（App Router互換性リスク + GSAP優位）。

### Q3: SSEストリーミング → Edge Runtime

- `export const runtime = 'edge'` + `export const dynamic = 'force-dynamic'`
- ReadableStreamパターンでレスポンス構築
- ケルト十字10枚は**段階的ストリーミング**（カード1枚ずつ解釈 → 総合解釈）

- 逆位置フレームワーク: **「エネルギーのブロック・内向化」を統一採用**
- 生成後に**人手レビュー必須**（5.6〜9.3h）、品質チェックリスト4軸を適用

### Q6: UX → タイプライター + 間 + アニメーションOFF

- カードフリップ後に**0.5〜1秒のワンテンポ間**（神秘感演出）

## 6. フォールバック・中止基準

### フォールバック戦略

| トリガー | フォールバック |
|---------|-------------|
| GSAPがNext.js App Routerで互換性問題（MVP初期2日以内に判定） | CSS @keyframes + Web Animations APIのみ（0KB追加） |
| ケルト十字SSEが300秒超過（LLM first-token-latency 15秒以上） | 2段階API分割（前半5枚 + 後半5枚 + 総合） |
| jsonrepair + retry後のJSON有効率が95%未満 | client.tsにnative Claude API (Structured Outputs) 直接切替オプション追加 |
| Vercel Hobby枠枯渇 | Cloudflare Workers移行（無制限実行時間 + Workers KV） |

### 中止基準

以下の**いずれかが成立**した場合にプロジェクト中止を判断:

1. 4層防御適用後もJSON出力安定性が**80%未満**
2. ケルト十字鑑定が300秒 + 分割でも**完了不能**
3. 小アルカナDBのハルシネーション率が**30%超**

いずれも「ローカルLLM + OpenAI互換API」のコア前提の破綻を意味する。Phase 1（MVP）完了前の早期判定を推奨し、**最大損失を32h以内に限定可能**。
