## 1. エグゼクティブサマリー

ロック済み設計（マルチエージェント + 段階推敲 + 対話ヒアリング + Claude Code サブエージェント実装）は、2026年時点の技術・コスト・品質研究の知見と**完全に整合**する。6視点の横断分析から、以下の構成が唯一の最適解として浮上した。

| 項目 | 結論 |
|------|------|
| **アーキテクチャ** | ハブ&スポーク + ファイルベース状態管理（非再帰制約のため唯一解） |
| **エージェント分割** | 4体構成（オーケストレーター / 執筆 / 批評 / 補強）|
| **推敲ステージ** | 3段階（草稿→批評→改稿）+ 外部クリティック |
| **20000字生成** | 章単位分割（6〜8章）+ ファイル結合方式 |
| **モデル配分** | Sonnet 4.6主軸（90%）+ Opus 4.6（10%：オーケストレーター） |
| **収束制御** | critic 70/100 OR 3反復（多層ハードキャップ） |
| **想定コスト** | $2.5〜$3.0/本（プロンプトキャッシング前提） |

## 3. 視点別主要知見

### 3.1 technical（技術的実現性）

- **非再帰制約は確定仕様**: サブエージェントは他のサブエージェントを Task 起動不可（issue #4182, #19077 他）
- **状態引き渡しはファイルベース推奨**: オーケストレーター（メイン会話）が Task ツールで専門エージェントを1段ネスト呼び出し。エージェント間は Markdown/JSON ファイル経由
- **Claude Code 出力キャップ ~32Kトークン**（issue #10738）: 20000字 ≒ 30〜40Kトークンのため章単位分割が必須級
- **モデル選択のティア設計**: Opus 4.6 = オーケストレーター、Sonnet 4.6 = 執筆・批評、Haiku 4.5 = 抽出・分類
- **tools フィールドはハード許可リスト**: Bash を許可すると sed/cat/ls 経由で制限迂回される（issue #21697）。読み取り専用は `[Read, Grep, Glob]` に絞る

### 3.2 cost（コスト・リソース）

| 構成 | コスト / 本 |
|------|------|
| Sonnet 4.6 × 3回推敲（キャッシュなし） | $1.91 |
| 同上 + プロンプトキャッシング | $1.72 |
| 同上 + Batch API + Cache | $1.40 |
| Opus 4.6 × 3回推敲 | $3.18 |
| ハイブリッド（Opus構成1回 + Sonnet草稿2回） | $2.33 |

- **推敲回数の最適点**: 研究では1〜2回で大部分の改善を獲得、GPT-4でも3回目で効果消滅
- **エージェント分割の落とし穴**: 「全文渡し」は線形倍増。差分/要約パス方式 + プロンプトキャッシングで2エージェント目以降の入力90%削減
- **Opus/Sonnet 価格差は1.67倍に縮小**（旧世代の5倍差から圧縮）
- **ヒアリング質問数はコスト無影響**: 全体の5%未満、UXが支配変数

### 3.3 risk（リスク・失敗モード）

| リスク | 対策 |
|--------|------|
| 推敲ループの平板化・統計的中央値収束 | divergence-first指示、最初の5案削除、マルチエージェント討論 |
| 20000字水増し・冗長化 | 構造（章数×節数）で制御、frequency_penalty、数値直接指定回避 |
| エージェント間トーン不整合（MASF Misalignment） | 共有スタイルガイドを全システムプロンプトに注入、order決定（outline→planning→writing） |
| 引用ハルシネーション（自己啓発で14〜94%） | **「具体的研究・人物名の引用禁止、一般原則として記述」制約が根本解** |
| 対話ヒアリング離脱 | 3問（最大5問）、事前宣言、進捗インジケーター |
| ループ暴走 | 外部強制の反復上限 + スコア閾値 + 繰り返し検出の多層化 |

### 3.4 alternatives（代替案比較）

| 観点 | 推奨 | 根拠 |
|------|------|------|
| 分割パターン | Writer-Critic基軸の3〜4体 | Cross-Refine 研究実績最多・コスト効率最良 |
| 推敲ステージ | 3〜4段階 + 外部クリティック | 段階数に対して品質向上は対数的 |
| ヒアリング方式 | 段階的深掘り（プログレッシブディスクロージャー） | エンゲージメントと情報品質の最大化 |
| オリジナリティ機構 | 体験ベース変換（ユーザー体験注入） | E-E-A-T適合、実流入1.5倍事例 |
| 文字数配分 | PREP反復型 + Pyramid Principle | 情報型ブログに最適、スキミング対応 |

### 3.5 content_quality_methodology（文章品質メソドロジー）

- **構成テンプレート移植性**: PREP法とPyramid Principleが最高、起承転結・序破急は品質要所のLLM生成が不安定
- **jReadability 指標（学術的に確立）**:
  スコア = 11.724 − 0.056×平均文長 − 0.126×漢語率 − 0.042×和語率 − 0.145×動詞率 − 0.044×助詞率
  → **漢語率（-0.126）と動詞率（-0.145）が読みやすさを最も下げる**
- **オリジナリティ分担**: 逆張りペルソナ / 異分野アナロジー変換 / Q&A挟み込み / 体験統合を専用エージェントに分担
- **陳腐化検出3層**:
  1. ブラックリスト（「いかがでしたでしょうか」「結論として」「〜しよう」等）
  2. perplexityスコア閾値
  3. 根拠チェックゲート（断言→「根拠・具体例は何か」）

### 3.6 agent_orchestration_patterns（オーケストレーション実装）

- **ハブ&スポーク + ファイルベース引き渡し**が唯一の堅実解
- **description自動委譲は信頼性が低い**: コミュニティ報告で「auto-selection remains unreliable」。**@-mention による明示指定**が推奨
- **Writer-Critic プロンプト構造**:
  - Writer: `Your task is to [goal]. Output your result to [file].`
  - Critic: `Review the output. Output JSON {strengths:[], weaknesses:[], suggestions:[]}. Do not modify original.`
- **収束判定は複合実装推奨**: (1)固定上限 + (2)スコア閾値 + (3)差分量チェック + (4)Stop Hook ゲート。LLM単独判定は思考縮退リスク

## 4. 視点間の矛盾と解決

| 衝突視点 | 要旨 | 解決策 |
|----------|------|--------|
| alternatives ↔ risk | 体験ベース変換推奨 vs 引用ハルシネーション警告 | **ユーザー自身の体験**は一次情報として採用、**第三者の研究・著名人発言**は具体指名禁止で抽象表現 |
| cost ↔ alternatives | 3〜4体が実用範囲 vs 役割特化4体が最高品質 | **4体構成で妥協**（オーケストレーター/執筆/批評/補強）。5体以上は自動委譲信頼性低下が品質向上を相殺 |
| technical ↔ cost | 章単位分割優位 vs 単一文書推敲前提コスト試算 | **章単位 + 差分パス方式**で整合。6〜8章×4000〜5000トークン、プロンプトキャッシングで入力重複90%削減 → $2.5〜$3.0/本 |
| agent_orchestration ↔ technical | 自動委譲信頼性低 vs 自然な1段ネスト前提 | **@-mention 明示指定による決定論的フロー**。手順書としてマークダウン化 |

## 5. 推奨アーキテクチャ（Primary）

### 5.1 構成

```
(A) オーケストレーター（メインセッション・Opus 4.6）
      │
      ├─► (B) 対話ヒアリングエージェント（Sonnet 4.6）
      │         └─ 3問（最大5問）段階的深掘り・事前宣言
      │
      ├─► (C) 執筆エージェント（Sonnet 4.6）
      │         └─ 章単位4000〜5000トークン
      │         └─ PREP反復型 + Pyramid Principle
      │
      ├─► (D) 批評エージェント（Sonnet 4.6）
      │         └─ 外部クリティック・JSON採点
      │
      └─► (E) 補強エージェント（Sonnet 4.6）
                └─ 体験統合・陳腐化置換・根拠チェック
```

### 5.2 実装詳細

| 項目 | 仕様 |
|------|------|
| ディレクトリ | `content/<slug>/` に `draft.md`, `review.json`, `outline.md`, `hearing.json` |
| YAMLフロントマター | `title / tags / audience / tone / word_count` |
| 章構成 | 6〜8章（序論10% / 本論75% / 結論15%） |
| 収束判定 | critic平均 ≥70/100 OR 3反復（外部強制ハードキャップ） |
| jReadability制約 | 1文40-60字、漢語率≤20%、文末変化 |
| 陳腐化ブラックリスト | 「いかがでしたでしょうか」等 + 具体引用禁止 |
| tools権限 | 執筆/補強=`[Read, Write, Edit, Glob]`、批評=`[Read, Grep, Glob]`（Bash除外）|
| オーケストレーション | @-mention 明示指定、description自動委譲非依存 |

### 5.3 想定リスク

- **章間一貫性劣化**: スタイルガイドファイル共有 + 最終統合推敲で整合性チェック
- **ヒアリング体験素材の質依存**: 体験談なし代替プロンプト（対立視点・異分野アナロジー）搭載
- **批評エージェントの自己バイアス**: 必要時に批評のみ Opus 4.6 に昇格オプション
- **文字数逸脱**: 各章後にカウント、±10%超過で補強エージェントが加筆/圧縮
- **均質化問題**（Science Advances 2024）: divergence-first指示をテーマごとにランダム化

## 6. フォールバック戦略

### 6.1 縮退構成（Fallback）

**3エージェント × 単一文書推敲 簡素版**
- オーケストレーター = ヒアリング + 構成設計兼務（Opus 4.6、事前質問3問）
- 執筆エージェント = 章単位追記（Sonnet 4.6）
- 批評エージェント = 全体レビュー + リライト指示（Sonnet 4.6）
- 補強は執筆に統合、推敲ループは最大2回

### 6.2 切り替えトリガー

初回3本生成テストで以下のいずれかが発生した場合:
1. 4体連携で章間トーン不整合が3本中2本以上
2. コストが$5/本を超過
3. 推敲3回でもスコア70/100未満が継続
4. description自動委譲失敗でオーケストレーション破綻
5. 実行時間が1記事30分を超過

## 9. 最終判定

**GO（Primary構成を採用）**

- ロック済み決定事項と6視点すべての最適解が一致する唯一の構成
- Writer-Critic 基軸・3段階推敲・4体構成・章単位分割・ファイルベース状態管理をすべて統合

### 9.1 撤退検討条件（参考）
