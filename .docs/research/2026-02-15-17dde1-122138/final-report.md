## 3. 6視点の調査結果

### 3.1 技術的実現性（technical）

**信頼度: High**

#### Claude Agents SDK統合

| 統合パターン | 概要 | 適合性 |
|--------------|------|--------|
| Proxy Pattern | BE経由でSDK呼び出し、BEがAPIキー管理 | 運用コスト負担大 |
| Direct Client (BYOK) | ユーザーが自分のAPIキーを使用 | 非技術系ユーザーの離脱リスク高 |
| Hybrid | 無料ユーザーはProxy、課金ユーザーはBYOK | 複雑だがバランス型 |

- ストリーミング：SSE（Server-Sent Events）が標準対応、実装容易
- セッション管理：SDK自体には含まれず、BE側でJWT+Redis等で実装が必要
- **課題：BYOKモデルは非技術系ユーザー（小説家・TRPG GM）にとって認知負荷が高い**

#### 世界観整合性担保の技術手法

| 手法 | 概要 | 信頼度 |
|------|------|--------|
| Multi-step Generation | 要素を順次生成し前段出力を次段に注入（7段階） | 高（WorldAnvil等で実績） |
| Structured Output (JSON Schema) | 形式的制約を強制 | 高（形式面のみ） |
| LLM-as-a-Judge | 矛盾検出→再生成 | 中（コスト高） |
| Few-shot Prompting | 整合性の高い例示を含める | 中 |

- 7要素一括生成は **最低7リクエスト、バリデーション含めば10-15回のAPI呼び出し** が必要
- JSON Schemaは形式面の制約のみ。「神Aの教義と戒律Bの矛盾」といった意味的矛盾は検出困難

#### 推奨技術スタック

| レイヤー | 第1候補 | 理由 |
|----------|---------|------|
| Frontend | Next.js (App Router) | SSE標準対応、Vercel AI SDK連携 |
| Backend | FastAPI | Python対応、非同期処理に強い |
| DB | PostgreSQL | RLS対応、Event Sourcing可 |
| Cache/Queue | Redis + BullMQ | ストリーミング接続管理 |

**主要ボトルネック：**
- LLM APIレート制限（Tier 3で4,000 RPM）
- WebSocket接続数（1サーバー約1,000接続が実用限界）
- DB全文検索（10万行超で遅延 → Elasticsearch移行が必要）

#### ギャップ

- BYOKのconversion rate低下の定量データが未公開
- Event Sourcing/CRDT導入工数がcost視点の見積もりに未反映（+200-400h）
- Claude Agents SDK以外のSDK（LangChain等）との比較未実施

#### 先行フィクション作品からの教訓

#### 整合性の自動評価手法

| 手法 | 精度 | 限界 |
|------|------|------|
| 知識グラフ + オントロジー検証 | LLM精度4.2倍向上の実績 | 形式的矛盾のみ（美的整合性は不可） |
| SCORE framework（物語整合性AI） | NCI-2.0で23.6%高整合性 | 長編向け、短い設定記述は未検証 |
| PMI（共起ベース） | 概念間の意味的整合性を定量化 | 宗教設定特化の適用例なし |

- **自己選択バイアス：** Reddit回答者はworldbuilding愛好家が過剰代表。穏健/無関心層が過少代表

- **モデレーター承認バイアス：** Reddit投稿は承認プロセスによる応答バイアスが発生
- **結論：** Reddit単独n=50でのWTP15%は **false positive（見かけの需要）を示す可能性が高い**

## 4. 視点間の矛盾と解決

### 4.1 cost × technical：コミュニティ機能の工数乖離

| | cost視点 | technical視点 |
|---|----------|--------------|
| フルMVP工数 | 280-420h | 480-820h（CRDT/OT含む） |

**解決：** コミュニティ機能はPhase B/Cでは非同期共有のみとし、リアルタイム編集は需要検証後に判断。非同期に限定すればcost視点の範囲内に収まる。

### 4.2 cost × alternatives：BYOKモデルの矛盾

- **cost視点：** BYOKはユニットエコノミクス解消策として合理的
- **alternatives視点：** 非技術系ターゲット層にとってBYOKはfriction（離脱原因）

**解決：** Freemium（月5回無料+$9.99/月）をデフォルト、BYOKをパワーユーザーオプションに限定。ただし **Freemiumのユニットエコノミクスは未検証**（後述のDA指摘MF-008）。

### 4.3 worldbuilding_coherence × technical：整合性の目標レベル

- **technical視点：** 矛盾排除（Multi-step + JSON Schema + LLM-as-a-Judge）
- **worldbuilding_coherence視点：** 完全整合性は不要、矛盾の出典明示がnarrative depthを生む

**解決：** worldbuilding_coherence視点が説得力で上回る。技術的検証手法を基盤としつつ、「矛盾の出典生成」機能を追加するハイブリッド設計を推奨。

### 4.4 user_intent_ethics × risk：技術的対策の十分性

- **user_intent_ethics視点：** IEEE 3原則の実装により倫理的境界の設計は可能
- **risk視点：** 技術的対策だけではリスク緩和不十分

**解決：** 両視点は相補的。IEEE 3原則は必要条件だが十分条件ではない。利用規約+通報メカニズム+定期的人的レビューの運用的対策を併用。コミュニティ機能（UGC共有）はPhase Cの需要検証後に限定導入。

### 4.5 alternatives × cost：市場不在下のコスト効率

- **alternatives視点：** 「10倍価値」の定量的回答なしに市場参入は危険
- **cost視点：** 段階的リリースとインフラコストの低さから参入可能

**解決：** alternatives視点が優先。コスト効率の高さは市場不在の場合に無意味。需要検証なしの開発投資は前回d-20260214-003447と同じ過ちを繰り返す。

## 6. 推奨アクション

### 6.1 Primary推奨：CONDITIONAL GO — 3段階Gate方式

```
Gate 0 → Gate 1 → Gate 2
（各GateでGO/NO-GO判定、不通過で即時ABORT）
```

#### Gate 0：需要仮説検証（2-4週間、20-40h、$500-1,500）

| 項目 | 内容 |
|------|------|
| **ターゲット再定義** | 「宗教創設志向者」→「小説家・ゲームデザイナー・TRPG GM・ワールドビルダーのうち宗教・信仰体系設定に課題を持つ層」 |
| **成果物** | Landing page（Carrd/Framer） + 機能コンセプト動画（3-5分） |
| **調査チャネル** | Reddit r/worldbuilding + r/writing + Discord servers + Twitter/X + 既存ツールフォーラム（合計n≧200、各チャネルn≧30） |
| **調査項目** | 利用意向（5段階）、WTP（$0-30）、現在のワークフロー・課題、汎用LLM利用経験、BYOKvsFreemium選好 |
| **GO条件** | WTP≧$5が全サンプルの10%以上（95%CI下限5%以上）かつ 利用意向4-5が30%以上 |
| **NO-GO条件** | WTP≧$5が5%未満、または利用意向4-5が15%未満 |

#### Gate 1：最小プロトタイプ検証（8-12週間、160-220h、$8,000-16,000）

| 項目 | 内容 |
|------|------|
| **前提** | Gate 0 GO判定後にのみ進行 |
| **プロトタイプ** | 認証あり・保存あり・テンプレートベース+LLM詳細化のハイブリッド |
| **スキーマ** | スマート7次元モデルベース、次元間依存関係の明示的モデル化 |
| **収益モデル** | Gate 0結果に基づき決定。Freemium（月5回無料+$9.99/月）がデフォルト |
| **倫理対策** | AI-generated表示義務、利用規約の法的レビュー、実在宗教模倣チェック（簡易版） |
| **GO条件** | 2週間で100名以上利用 + D7継続率>20% + NPS>30 + 有料転換率>2% |
| **NO-GO条件** | 利用者50名未満、またはD7継続率<10%、またはNPS<0 |

#### Gate 2：AI統合MVP検証

- Gate 1 GO判定後にのみ進行
- スコープ・工数はGate 1の実測データに基づき決定
- コミュニティ機能はGate 2の需要シグナルに基づき判断

### 6.2 Fallback推奨：オープンソース・プロンプトテンプレート集

| 項目 | 内容 |
|------|------|
| **トリガー** | Gate 0でNO-GO判定、またはGate 0実施困難（n=200の50%未満） |
| **成果物** | スマート7次元モデルベースの構造化テンプレート（Markdown/YAML）+ 推奨プロンプト集 |
| **配布** | GitHub / Gumroad（無料/ドネーションベース） |
| **工数** | 40-80h（テンプレート設計20-40h + ドキュメント20h + サンプル生成10-20h） |
| **コスト** | $0-500 |
| **検証指標** | GitHub Stars>500、ダウンロード>1,000、Webアプリ化要望>20% |

**DA推奨：Gate 0とFallbackの並行実施。** Gate 0調査期間（4-8週間）中にテンプレート集を並行開発し、Gate 0結果に関わらずリリース。これにより(a) Gate 0投資の保険効果、(b) テンプレート集自体が追加の需要検証ツール、(c) Forge Harness Phase 2の軽量テストケースが確保される。

### 6.3 ABORT条件

以下のいずれかに該当する場合、全活動を中止：

1. Claude Agents SDKの利用規約がBYOKモデルを禁止していることが確認された場合
2. Gate 0で2回連続NO-GO判定、またはGate 1でNO-GO判定
3. 法的調査で宗教創設支援が規制対象となるリスクが高いと判明した場合
4. 架空宗教ピボット後も前回と本質的に同じ結論（事業的正当性の未証明）に至った場合
