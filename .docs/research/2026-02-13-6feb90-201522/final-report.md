## 2. 全視点一致事項

### 2.1 要件再定義の必要性（最重要発見）

**5視点中5視点が独立して同一結論に到達**:

> 現行の「完全自動リプライ・自動DM」要件はX APIポリシー違反リスクが高く、半自動化（HITL）+ スケジュールベース + 公式API準拠への調整が合理的。

| 視点 | 根拠 |
|------|------|
| Technical | X APIポリシーが「キーワード検索のみに基づく自動リプライ」を明示的に禁止。半自動化で公式API移行が可能 |
| Cost | 半自動化は完全自動化比で導入コスト大幅削減。承認フロー付きツールは月額250円/ユーザー程度 |
| Risk | 530万アカウント凍結（2024年前半）。Patchrightは行動分析には無力。公式API+HITL移行でリスク消滅 |
| Alternatives | HITL+スケジュールベース+公式API準拠でブラウザ自動化依存を大幅削減可能。ただし完全排除は困難 |
| Operational Resilience | キーワード単独の自動返信はX.comポリシー違反。時刻ベース/手動トリガーなら完全準拠 |

### 2.2 観測性基盤の完全欠如

- 現行はconsole.log/console.errorの非構造化ログのみ
- 操作成功率・レスポンスタイム・セレクタヒット率等の定量メトリクスが一切取得不可
- **改善効果の定量評価が不可能**な状態

### 2.3 メモリリーク — 「時限爆弾」

- `sentReplies` Mapが無制限に成長
- `seen-items.json`が肥大化
- Browser Pool上限未設定
- 長時間運用で「徐々に悪化し気づきにくい」典型パターン

### 2.4 bypassPermissionsのセキュリティリスク

- 既知CVE（CVE-2025-54794/54795）でコマンドインジェクションによるホワイトリスト完全バイパスが実証済み
- 「信頼されていないコンテンツを信頼された権限で処理」する根本的欠陥
- 半自動化移行で承認フロー導入が可能

## 5. 推奨アクション

### 5.1 Primary推奨：Phase 0差し戻し → 要件再定義

```
Step 0: 要件再定義の意思決定（人間判断必須）         ← 最優先
Step 1: 観測性基盤構築（Pino + Discord webhook）      4-8h ※
Step 2: メモリリーク修正（Map上限 + eviction policy）  2-4h ※
Step 3: 同期I/O移行（初期化sync許容 + ランタイムasync）4-8h ※
Step 4: テスト基盤構築（Vitest + ユニットテスト）     8-16h ※
Step 5: エラーハンドリング + graceful shutdown         4-8h ※
Step 6: 公式API移行 or モジュール分割                 16-40h ※

※ 全工数は業界平均ベースの推定値。実測データで要補正
```

**推奨する運用モデル**:

| 要素 | 現行 | 推奨 |
|------|------|------|
| リプライ・DM | 完全自動（ブラウザ自動化） | **半自動化**（LLM提案→人間承認→公式API送信） |
| 投稿 | 自動投稿（ブラウザ自動化） | **スケジュールベース**（時刻/手動トリガー） |
| API | なし（ブラウザ操作のみ） | **公式API Basic tier**（$100/月） |
| ブラウザ自動化 | 全操作 | **公式APIでは不可能な操作のみ**に限定 |

**要件再定義による効果**:

| 項目 | 改善内容 |
|------|----------|
| CQ-2（リプライ返信） | 公式API `in_reply_to_tweet_id` + HITL承認フローに置換。ブラウザ自動化不要 |
| CQ-4（メモリリーク） | ブラウザ長時間稼働不要でリスク大幅低減 |
| CQ-5（テスト戦略） | E2E対象がAPI呼び出しに変わりテスト容易性が劇的向上 |
| CQ-6（DM PIN認証） | 公式API移行でブラウザ認証フロー自体が不要 |
| CQ-7（耐障害性） | DOM変更リスク消滅。APIレート制限対応に問題移行 |

**Primary推奨のリスク**:

- 公式API Basic tier 月額$100+のランニングコスト発生
- 半自動化による応答速度低下（人間承認待ち）
- X API仕様・価格の頻繁な変更リスク
- DMレート制限（新規アカウント300-500/日）が運用規模を制約
- ブラウザ自動化の完全排除は困難（一部用途で依然必要）
- 要件再定義の判断に時間がかかりPhase 2着手が遅延するリスク

### 5.2 Fallback推奨：完全自動化継続

**発動条件**: 人間判断で「完全自動化要件は変更不可」と明示された場合

```
Step 1: 観測性基盤（Pino + メトリクス）              4-8h ※
Step 2: メモリリーク修正（Map上限 + eviction）        2-4h ※
Step 3: エラーハンドリング + PM2 + node-windows       4-8h ※
Step 4: 同期I/O移行                                   4-8h ※
Step 5: browser.jsモジュール分割（POM）               16-24h ※
Step 6: Vitest導入                                    8-16h ※
Step 7: セレクタ健全性 週次E2E + Discord通知          4-8h ※

※ 全工数は業界平均ベースの推定値。実測データで要補正
```

**構造的に残存するリスク**:
- X APIポリシー違反リスク
- アカウント凍結リスク（530万件/半年のペース）
- DOM変更による月10-15時間の保守コスト
- bypassPermissionsのセキュリティリスク

### 5.3 ABORT条件

以下のいずれかが判明した場合、プロジェクト継続を中止:

1. X.comがPatchright/Playwright系を完全ブロック、かつ公式API移行も不可能
2. アカウント凍結が実際に発生し、復旧不能
3. X API価格改定でBasic tierが$500/月超
4. 3視点以上が「根本的に破綻」と判定

**ABORT時の機会費用**: 実装投資の損失はあるが、LLMベースコンテンツ生成パイプラインの知見は他プラットフォーム（Bluesky/Mastodon等）に転用可能。

Step 0で完全自動化継続を選択 → 改善途中でX.comがPatchright系を完全ブロック → 投資工数（20-40h）が無駄 → アカウント凍結 → bypassPermissions経由でセキュリティ侵害。確率は低いが、構造的にはABORT条件の同時発生に相当。

### 6.5 Should-Fix（推奨改善）

| ID | 内容 |
|----|------|
| S-1 | Step 0待機中に並行実施可能な「要件非依存の改善」を明示すべき（メモリリーク修正、bypassPermissions対策） |
| S-2 | X API開発者ポータル承認の具体的条件（審査基準・所要期間・拒否率）の補足調査 |
