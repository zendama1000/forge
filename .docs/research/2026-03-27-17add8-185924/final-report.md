- **必要な独自実装**: 指数バックオフ＋ジッター付きリトライラッパー、レートリミット(429)検出、サーキットブレーカー

#### 認知ループの実態（前提修正）

SC調査計画ではOBSERVE/ORIENT/ACTの3フェーズと想定していたが、実コード分析により**既に7フェーズループが実装済み**:

```
PERCEIVE → REMEMBER → DECIDE → GENERATE → ACT → REFLECT → SLEEP
(~30行)    (~30行)    (~90行)   (~30行)    (~50行)  (~35行)   (~20行)
```

- `cognitive-loop.js`: 全体626行、メインループ本体約280行
- `decide()`: Claude API 1回呼出＋JSONパース＋フォールバック処理
- 課題は「DECIDEの新規追加」ではなく「**既存DECIDEの品質向上**」に再定義

#### ESM構成とテストFW互換性

- `package.json`: `type: module`（ESM専用）
- dynamic import: cognitive-loop.js内3箇所（オプショナル機能の遅延ロード）
- Vitest v4: **設定不要でそのまま導入可能**（ESMネイティブ）

### 3.3 リスク・失敗モード（Risk）

**信頼度: High** | **判定: 6領域全て「高リスク」**

#### R1: X/Twitter DOM変更・検出強化

- GraphQL `doc_id`は**2〜4週間ごとに予告なくローテーション**
- CSSクラス名はランダム文字列で、デプロイまたはセッション単位で変更
- Patchright検出スコア: CreepJS **67%**（100%回避ではない）
- 月10〜15時間の保守コストが必要
- **フォールバック**: マネージドサードパーティAPI（twitterapi.io等）、公式X API（$100/月〜）

#### R2: アカウントBAN・レート制限

- 2024年にXは**8億アカウントを停止**（Q1だけで464百万件がスパム関連）
- コールドDM自動送信は**ToSで明示的に禁止**
- 公式APIのDMレート制限: 1,440/24hr/app だが、実際には**1req/24hr/user**の報告あり

**安全ガードレール設計**:
- 新規アカウント: 15〜20 DM/日 上限
- 確立済みアカウント: 50〜75 DM/日 上限
- リンク禁止（特に短縮URL）、テンプレートメッセージ禁止
- 自動アクション比率 50%以下維持
- アカウント警告時: 制限50%削減 + 48〜72時間停止

#### R3: Claude Agent SDK breaking change

- SDK は **0.x 系（pre-1.0）**でSemVer保護なし
- 確認済みbreaking change:
  - v0.1.0: SDK名称変更（`@anthropic-ai/claude-code` → `@anthropic-ai/claude-agent-sdk`）
  - v0.2.68: サイレントbreaking change（effortデフォルト変更、CHANGELOGなし）
  - Claude 4.6: assistantプリフィルが事前警告なしに400エラー化
- **対策**: アダプターパターン（ACL 3層構造）で依存を抽象化

#### R4: 誤判断のロールバック不可能性

- DM送信・Reply投稿は**外部コミット済みアクション**であり、真のロールバックが不可能
- 10ステップチェーンで各95%精度 → 全体精度は**約60%**に低下
- **対策**: 不可逆アクション実行前のHITLチェックポイント必須

#### R5: スコープ肥大化

- 「全面改善」フレーミング自体が**最大のリスク要因**
- 自然な停止条件がなく、各改善点が新たな改善点を発見し続ける
- **対策**: 明示的DoD + 時間ボックス + RICEフレームワーク + 接線的変更テスト

#### R6: flaky test問題

- Playwrightのflaky根本原因: レースコンディション、不安定なセレクタ、ネットワーク不確実性、テスト間状態汚染、CI環境差異
- **対策**: data-test属性、テストごとの独自ブラウザコンテキスト、外部呼出モック化、知的テスト選択

### 3.5 認知ループの行動モデル設計（Cognitive Architecture）

**信頼度: Medium〜High**

#### CA1: エネルギーカーブモデル

線形枯渇モデル（E(n) = E0 - k*n）は**科学的に否定済み**。Central Governor Modelに基づく多入力制御シグナルを推奨:

```
DECIDE_energy(t) = f(current_load, expected_future_demand, goal_importance, opportunity_cost)
```

**実用的簡略版（初期実装推奨）**:
1. **時間帯重み**: 3区分（高活動/通常/低活動）
2. **アクション累積カウンタ**: 20-40アクション/セッション閾値
3. **コンテンツ種別乗数**: engage/rest/skip の3分類

ボット検出回避の観点から、自然な行動分散のモデル化が重要。

#### CA2: フェーズ間データフロー

各フェーズ間の受渡しデータは3クラス:
1. **新規環境入力** (O_t): 生観測値
2. **前回推論トレース** (Z_(t-1)): 方向付け/決定の根拠
3. **実行フィードバック** (E_(t-1)): 前回Actの結果

**推奨アーキテクチャ**:
- 短時間ループ（<秒）: インプロセス状態管理
- 長時間ループ（>分）: 非同期クロンワーカー + SQLite/KVストア
- 状態永続化: エピソードストリーミング（全記録）+ サマリー圧縮のハイブリッド
- **必須要件**: ActからObserveへのフィードバックループのクローズ

#### CA3: 判断品質の定量評価

4柱フレームワーク:

| 柱 | 指標 |
|----|------|
| LLM品質 | 指示遵守スコア、ポリシー準拠率、安全違反数 |
| メモリ品質 | 検索Precision/Recall/F1、更新正確率 |
| ツール品質 | ツール分類精度、パラメータ精度、シーケンス正確性 |
| 環境適応 | ガードレール違反数、タスク完了率 |

- **Action Advancement Score**: 0〜1/ステップ（>0.7:進展、<0.3:停滞）
- **pass@k vs pass^k**: ベストエフォート能力 vs 一貫性を区別

#### CA4: 複数アカウント運用

**推奨**: ハイブリッド分離アーキテクチャ
- 共有推論エンジン + アカウント別独立セッション/メモリ/認証情報ストア
- Intrinsic Per-Account Memory: **38.6%トークン効率向上**（実証済み）
- Named Connection Binding: 認証情報の構造的バインドが**セキュリティ上必須**
- 共有ナレッジはツール経由のみで注入（read-only共有ストア）

### 3.6 AIエージェント+ブラウザ自動化のテスト戦略（Test Strategy）

**信頼度: Medium〜High**

#### TS1: テスト層の最適比率

| 層 | 比率 | 役割 | ツール |
|----|------|------|--------|
| ユニット/モック | 60-70% | 決定論的基盤 | Vitest + 固定JSONモック |
| MSW統合 | 20-30% | HTTP傍受・ハンドラ共有 | MSW + playwright-msw |
| 実ブラウザE2E | 10-20% | クリティカルフロー検証 | Playwright標準（**Patchright非使用**） |

**重要**: Patchrightは外部サービス（X/Twitter）向けに限定し、テスト時はPlaywright標準を使用すべき。

#### TS2: LLMレスポンスのモック戦略

| 戦略 | 用途 | CI適性 |
|------|------|--------|
| **固定JSONモック** | オーケストレーションロジック・エラーハンドリング | 最適（決定論的・ゼロコスト） |
| スナップショット | **実質不適用**（LLM非決定性で誤検知だらけ） | 非推奨 |
| **LLM-as-Judge** | 意味的品質（忠実性・関連性）評価 | オンデマンド（コスト大） |

最適構成: **固定JSONモック（CI基盤） → レコード&リプレイ（統合層） → LLM-as-Judge（品質ゲート）**

#### TS3: 非決定的認知ループのテスト

- **プロパティベーステスト（PBT）**: 不変条件検証に有効。Anthropicが100パッケージで実バグ発見（$5.56/バグレポート）
- **ファジング**: エージェント専用ファジャー（Reaper等）は新興分野
- **実践的アプローチ**:
  - pass@k / pass^k 二指標（Anthropic推奨）
  - オーケストレーション層を決定論的に保ち、AI非決定性を境界付きユニットに隔離
  - ゴールデントラジェクトリ（成功トレース）による行動回帰テスト

#### TS4: 5-10%→目標カバレッジの最短経路

**テスト対象の優先順位**:
1. コアユーザーワークフロー（ログイン・主要操作）— 障害コスト最大
2. 欠陥再発頻度の高いモジュール
3. 外部連携ポイント（APIクライアント・DBアダプタ）
4. 高複雑度ビジネスロジック
5. セキュリティ・認証
6. 新規コードへのカバレッジゲート適用（80%+）で負債蓄積防止
